# ETC Platform v2.1 — From Skeleton to Working System

**Date:** 2026-02-27
**Status:** Draft
**Author:** Jason Vertrees + Claude
**Input:** v2 PRD, v1 lessons learned, v2.0 implementation audit (256 tests, 14 modules)

---

## Executive Summary

v2.0 built the **orchestration substrate** — Postgres state, event bus, phase engine, execution graphs, agent runtime stubs, SEM decision loop, guardrail framework. 256 tests confirm the bones are solid.

v2.1 turns the skeleton into a working system by completing three things:

1. **Agent tools that actually work** — agents can search source materials, share knowledge, and have their outputs checked against domain truth
2. **Topology builder** — the SEM can design multi-layer execution graphs from source material inventory, not just execute manually-designed ones
3. **Human-in-the-loop gates** — humans approve topologies before expensive fan-outs, override guardrail violations with justification, and resolve knowledge conflicts

**Success criterion:** `etc init acme-corp --type re-engineering` → intake interview → topology generated → human approves → agents execute → guardrails enforce → synthesis completes → `etc status` shows it all. Across sessions, across restarts.

## v1 Lessons That Drive v2.1 Priorities

Each priority below maps to a specific v1 failure mode. This isn't a wish list — it's the minimum to avoid repeating known failures.

| v1 Failure | v2.1 Fix | Priority |
|------------|----------|----------|
| **Volume trap** — researchers drowned in implementation artifacts | Wire `search_files` to source_materials with priority filtering | P0 |
| **Platform artifacts as domain truth** — Salesforce patterns reproduced | Domain briefing injection + domain fidelity guardrail | P0 |
| **No cross-agent knowledge sharing** — conflicts found post-hoc | Wire `query_knowledge`/`contribute_knowledge` agent tools to knowledge.py | P0 |
| **Flat fan-out insufficient** — Acme Corp needed 3 layers | Topology builder: SEM-driven recursive decomposition | P1 |
| **SEM context erosion** — design rationale lost to compaction | Already fixed by C2 (stateless SEM) — but add decision audit trail | P1 |
| **Guardrails advisory, not enforced** — "MANDATORY" in prompt ignored | Domain fidelity check as middleware + violation → retry context injection | P0 |
| **Manual topology design** — human wrote Acme Corp research plan | Topology builder + human approval gate before execution | P1 |
| **No HITL gates** — no approval before expensive parallelization | Topology review, guardrail override, conflict resolution CLI | P1 |

## Roadmap: Four Waves

### Wave 1: Agent Tools (P0 — Agents Can Actually Work)

**Goal:** Agents deployed by the SEM can search source materials, read/write knowledge, and have their outputs validated against domain truth.

#### Task 1.1: Wire Agent Knowledge Tools

**What:** Replace the `query_knowledge` and `contribute_knowledge` stubs in `agent_runtime.py` with real implementations that call `knowledge.py`.

**Current state:**
```python
# agent_runtime.py:212-223 — STUBS
@agent.tool
def query_knowledge(ctx: RunContext[AgentDeps], key: str) -> str:
    return f"No knowledge entry found for key: {key}"  # <- stub

@agent.tool
def contribute_knowledge(ctx: RunContext[AgentDeps], key: str, value: str) -> str:
    return f"Knowledge entry recorded: {key}"  # <- stub
```

**Target:** Tools call `knowledge.query_knowledge()` and `knowledge.contribute_knowledge()` using `ctx.deps.conn` and project_id (resolved from node → graph → project chain). Returns real knowledge entries or writes real entries to Postgres.

**Design consideration:** The agent tool receives `value: str` but `knowledge.contribute_knowledge` expects `value: Any` (stored as JSONB). The tool should accept a string and attempt JSON parse, falling back to wrapping in `{"text": value}`.

**Requires:** Project ID available in AgentDeps (currently has conn, node_id, run_id, assignment — needs project_id added).

**Tests:**
- Agent tool `query_knowledge` returns real entries from knowledge_entries table
- Agent tool `contribute_knowledge` creates real entries in knowledge_entries table
- Agent tool `contribute_knowledge` supersedes existing entries with same key
- Project ID correctly resolved from node → graph → project chain

#### Task 1.2: Wire Agent Search Tool to Source Materials

**What:** Replace the `search_files` stub with a real implementation that queries the `source_materials` table, filtered by priority.

**Current state:**
```python
# agent_runtime.py:205-209 — STUB
@agent.tool
def search_files(ctx: RunContext[AgentDeps], pattern: str) -> list[str]:
    return []  # <- stub
```

**Target:** Queries `source_materials` for the project, filtering by name/type/classification matching the pattern. Returns structured results including `name`, `type`, `classification`, `priority`, `path`, and `reading_instructions`. Respects priority ordering (primary > high > medium > context_only).

**Design:** The tool should support two modes:
1. **Inventory mode** (pattern = `"*"` or empty): List all source materials for the project, ordered by priority
2. **Search mode** (pattern = keyword): Filter by `name ILIKE '%pattern%'` or `classification = pattern`

**Tests:**
- Returns source materials matching name pattern
- Orders results by priority
- Includes reading_instructions in results
- Returns empty list for project with no materials

#### Task 1.3: Domain Briefing Injection into Agent Prompts

**What:** When deploying an agent, automatically inject the project's domain briefing, classification, and anti-pattern catalog into the system prompt — guaranteed by the platform, not by agent self-discipline.

**Current state:** `load_agent_prompt()` accepts an optional `context` dict but it's not populated with domain-specific content during `AgentRunner.deploy()`.

**Target:** `AgentRunner.deploy()` queries the project's classification and source materials to build a mandatory context block:
```
=== MANDATORY CONTEXT ===
Project Type: re-engineering
Domain Briefing: [from primary source materials with reading_instructions]
Anti-Pattern Catalog: [if re-engineering: boolean flags, hardcoded enums, legacy mappings]
```

This context is prepended to the agent's system prompt by the platform — the agent cannot skip it.

**Design considerations:**
- Context must fit within a reasonable token budget (configurable, default 4000 tokens)
- Primary and high-priority materials get full reading_instructions; medium gets name only; context_only is excluded
- For re-engineering projects, the anti-pattern catalog from `guardrails.py` patterns is included as natural language

**Requires:** Project ID in AgentDeps (same as Task 1.1).

**Tests:**
- Re-engineering project injects anti-pattern catalog
- Primary materials include reading_instructions
- Context-only materials excluded from prompt
- Total context respects token budget

#### Task 1.4: Domain Fidelity Guardrail

**What:** Implement the `domain_fidelity_check` guardrail rule — the most important guardrail from v1, the one that would have caught the OPA/Rego incident and the Salesforce pattern reproduction.

**Current state:** `guardrails.py` has `AntiPatternScanRule` (regex-based) and `OutputSchemaValidationRule`. No domain fidelity check.

**Target:** New `DomainFidelityRule` that:
1. Loads domain axioms from the project's primary source materials
2. Checks agent output for contradictions against those axioms
3. Uses a lightweight LLM call (configurable model, default haiku) for semantic comparison
4. Returns violation details with specific axiom references

**Design:**
- Domain axioms are extracted from source materials with `classification = 'domain_truth'` or `priority = 'primary'`
- The check is a short Claude API call: "Given these domain axioms: [...]. Does this agent output contradict any of them? If yes, list which axioms are violated and how."
- Severity: critical (violations block acceptance)
- Configurable: can be disabled per project via config

**Architecture decision:** This is the first guardrail that requires an LLM call, not just regex. It establishes the pattern for "smart" guardrails. Keep it simple — one focused API call per output, not a conversation.

**Tests:**
- Output that contradicts domain axioms fails with specific violation details
- Output consistent with domain axioms passes
- Skipped when no domain axioms are configured
- Returns violation details referencing specific axioms

---

#### Task 1.5: Guardrail → Retry Context Injection

**What:** Wire the guardrail violation flow: when a guardrail fails, the violation details are injected into the retry context so the re-spawned agent knows what went wrong.

**Current state:** `retry.py`'s `execute_retry()` accepts `violation_details: str` and injects it into the assignment. But nothing in the system actually calls `execute_retry` with violation details from guardrail results.

**Target:** After `GuardrailMiddleware.check_and_record()` runs on an agent output:
- If critical failure → emit `GUARDRAIL_VIOLATION` event with violation details
- SEM receives event → calls `execute_retry(node_id, violation_details=details_from_guardrail)`
- Re-spawned agent gets violation context in its assignment

**Integration points:**
- `agent_runtime.py` → `guardrails.py`: After recording output, run guardrail checks
- `guardrails.py` → `events.py`: Emit GUARDRAIL_VIOLATION on critical failure
- `orchestrator.py` → `retry.py`: SEM handles GUARDRAIL_VIOLATION by calling execute_retry

**Tests:**
- Critical guardrail failure emits GUARDRAIL_VIOLATION event
- SEM decision on GUARDRAIL_VIOLATION triggers retry
- Retried agent receives violation details in assignment
- Non-critical failures don't trigger retry (emit warning event only)

### Wave 2: Topology Builder (P1 — SEM Can Design, Not Just Execute)

**Goal:** The SEM can assess source materials, design a multi-layer execution graph, and present it for human approval — automating what was done manually for Acme Corp.

#### Task 2.1: Topology Builder Module

**What:** New `topology.py` module that generates execution graphs from source material inventory + project classification.

**The recursive algorithm (from PRD §6.2):**
```
function buildGraph(scope, depth=0):
  if scope fits in one agent → return LeafNode(agent_type, assignment)
  partitions = decompose(scope) by dimension
  children = []
  for each partition:
    children.append(buildGraph(partition, depth+1))
  if children need synthesis:
    reduce = ReduceNode(reads outputs of children)
    return CompositeNode(children + [reduce])
  return CompositeNode(children)
```

**Design:** Two-stage process:
1. **Assessment stage:** Call Claude with project classification + source material inventory → structured output: `TopologyPlan` (dimensions, partition strategy, estimated depth, estimated agent count)
2. **Graph generation stage:** Convert TopologyPlan into concrete execution graph via `graph_engine.py` API calls

**Structured output model:**
```python
class PartitionStrategy(BaseModel):
    dimension: str  # e.g., "bounded_context", "cx_workflow", "migration_concern"
    items: list[str]  # e.g., ["compliance", "vendor_management", "insurance"]
    agent_type: str  # e.g., "researcher"

class TopologyPlan(BaseModel):
    layers: list[list[PartitionStrategy]]  # layer 0 = first fan-out, etc.
    reduce_strategy: str  # "single_synthesis" | "sub_synthesis_then_final"
    estimated_agents: int
    reasoning: str
```

**Tests:**
- Assessment produces valid TopologyPlan from source material inventory
- Graph generation creates correct node structure with dependencies
- Reduce nodes depend on all leaf nodes in their layer
- Multi-layer topologies have correct inter-layer dependencies

#### Task 2.2: SEM Topology Design Decision Type

**What:** Add `DESIGN_TOPOLOGY` to the SEM's `DecisionType` enum, with corresponding execution logic.

**Current state:** `orchestrator.py` has 6 decision types: DEPLOY_AGENT, ADVANCE_PHASE, CHECK_DOD, WAIT, REQUEST_HUMAN_INPUT, MARK_NODE_READY.

**Target:** New `DESIGN_TOPOLOGY` decision type that the SEM uses when:
- The project is in Decompose phase
- No execution graph exists yet
- Source materials are triaged

**Execution:** Calls topology builder → stores generated TopologyPlan in events table → emits TOPOLOGY_DESIGNED event → waits for human approval before creating the execution graph.

**Tests:**
- SEM produces DESIGN_TOPOLOGY when in Decompose phase with no graphs
- Topology is stored as event payload
- Human approval required before graph creation

#### Task 2.3: Dynamic SEM Prompt Construction

**What:** Make the SEM system prompt project-aware by injecting classification, domain context, and known conflicts.

**Current state:** `SEM_SYSTEM_PROMPT` is a static string constant. `load_scoped_state()` provides phase, DoD, graphs, and ready nodes — but no domain context.

**Target:** `load_scoped_state()` additionally returns:
- `project_classification`: from projects table
- `source_material_summary`: count by type/priority from source_materials
- `unresolved_conflicts`: from `knowledge.detect_conflicts()`
- `anti_pattern_catalog`: if re-engineering project

The user prompt (`_build_user_prompt()`) includes these in a `=== Project Context ===` section.

**Tests:**
- Re-engineering projects include anti-pattern catalog in SEM prompt
- Unresolved conflicts appear in SEM context
- Source material summary included in state

### Wave 3: Human-in-the-Loop (P1 — Humans Gate Expensive Decisions)

**Goal:** Humans approve topologies, override guardrails, and resolve knowledge conflicts through the CLI.

#### Task 3.1: Topology Approval Flow

**What:** Before the SEM creates an execution graph from a TopologyPlan, the human must approve it via `etc topology review`.

**CLI commands:**
```bash
etc topology show              # Display the proposed topology (tree diagram)
etc topology approve           # Approve → SEM creates execution graph
etc topology reject "reason"   # Reject → SEM re-generates with feedback
```

**Design:** Uses Rich to render a tree diagram of the proposed topology: layers, nodes, agents, dependencies. The human sees exactly what will be deployed before any API calls are made.

**Tests:**
- `etc topology show` renders tree from TopologyPlan
- `etc topology approve` creates execution graph via graph_engine
- `etc topology reject` stores feedback, SEM can re-generate

---

#### Task 3.2: Guardrail Override Flow

**What:** When a guardrail blocks acceptance, a human can override it with justification.

**CLI commands:**
```bash
etc guardrails status                    # Show all guardrail results for current phase
etc guardrails override <check_id> "reason"  # Override with justification
```

**Design:**
- Override records the human's justification in `guardrail_checks` table (new `override_reason` column, `overridden_by`, `overridden_at`)
- Override changes `passed` to `true` on that specific check
- Re-evaluates agent_output acceptance after override
- Emits HUMAN_RESPONSE event with override details

**Tests:**
- Override changes check.passed to true
- Agent output re-evaluated after override
- Override reason stored in database
- Override emits event for audit trail

#### Task 3.3: Knowledge Conflict Resolution CLI

**What:** CLI for discovering and resolving knowledge conflicts between agents.

**CLI commands:**
```bash
etc knowledge list                    # Show all knowledge entries
etc knowledge conflicts               # Show entries where agents disagree
etc knowledge resolve <key> <winner_id>  # Resolve conflict by picking winner
```

**Design:** `etc knowledge conflicts` calls `knowledge.detect_conflicts()` and renders a Rich table showing each conflict: key, contributor agents, their values, timestamps. Human picks a winner.

**Tests:**
- Conflicts displayed with contributor details
- Resolution supersedes losing entries
- Event emitted for conflict resolution

### Wave 4: Polish + Remaining Guardrails (P1/P2)

#### Task 4.1: Decision Audit Trail

**What:** Store every SEM decision with its full reasoning in a queryable format. `etc history` shows the decision log.

**Design:** Each `execute_decision()` call already emits events. Enhance to also store the full `SEMDecision` object (decision_type, reasoning, parameters) in the event payload. Add `etc history` CLI command that renders the decision timeline.

---

#### Task 4.2: Coverage Gate Guardrail

**What:** `coverage_gate` rule that checks test coverage meets a project-configurable threshold (default 80%).

**Design:** Runs `pytest --cov` on the project directory, parses the output, compares against threshold. Only applicable to `output_type = "code"`.

---

#### Task 4.3: Security Scan Guardrail

**What:** `security_scan` rule that checks generated code for basic OWASP patterns.

**Design:** Regex-based scan for common vulnerabilities: SQL injection (string concatenation in queries), XSS (unescaped output), hardcoded secrets, insecure deserialization. Severity: high.

---

#### Task 4.4: TDD Verification Guardrail

**What:** `tdd_verification` rule that checks whether tests were written before implementation.

**Design:** Analyzes git history for the files in the agent's output. If test files have commits after implementation files, violation. Severity: medium.

## Success Criteria (The Acme Corp Test)

v2.1 is done when:

1. `etc init acme-corp --type re-engineering` → interactive intake populates source materials
2. `etc run` in Decompose phase → SEM calls topology builder → multi-layer plan generated
3. `etc topology show` → human sees the plan → `etc topology approve`
4. `etc run --auto` → Layer 1 agents deployed in parallel
5. Each agent output runs through guardrails including domain fidelity check
6. Critical guardrail violation → agent re-spawned with violation context
7. Agents share knowledge in real-time via `query_knowledge`/`contribute_knowledge`
8. `etc knowledge conflicts` → human resolves disagreements
9. Layer 1 completes → Layer 2 auto-deploys → synthesis agent runs
10. `etc status` shows everything, `etc history` shows every decision
11. Close laptop. Reopen. `etc status` shows exactly where things stand.

## Relationship to Vision

The v2.1 roadmap implements the **first practical instance** of the broader vision described in `VISION.md` — "Industrialized System Synthesis." Specifically:

- **Declarative reconciliation** starts with the SEM's stateless decision loop (change the execution graph, the system converges)
- **Semantic queryability** starts with the knowledge graph (agents query structured facts, not probabilistic guesses)
- **Federated spec topology** starts with source material triage (multiple source types composed into a unified domain understanding)
- **Agent swarms as gene expression** is exactly what the execution graph does (agents spun up to reconcile state, not permanent residents)

v2.1 is the minimal viable instance of this architecture — one where the genotype (PRD + source materials + domain briefing) drives the phenotype (working software) through controlled agent expression, with guardrails as the selection pressure.
