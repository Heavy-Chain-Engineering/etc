# ADR-F-2026-05-26-002: Completeness is enforced by a per-cell matrix walk, not by prose reasoning

**Date:** 2026-05-26
**Status:** Accepted

**Context:** A production bug shipped unindexed queries because nothing forced the architect to consider query performance at the data-access layer. The root failure was not ignorance — it was that the quality consideration was *implicit* and therefore skippable under time pressure. The design must make the consideration un-skippable. Research into established practice (arc42 §8 crosscutting concepts, ATAM scenario analysis, AWS Well-Architected per-pillar questions, STRIDE-per-element) converged on a single structural insight: completeness is enforced by a **two-axis walk** — (architectural layer) × (quality criterion) — where each applicable cell demands an explicit answer. STRIDE does not let a reviewer forget "tampering on the data store" because the table *has that cell*; the per-cell discipline is the forcing function, not the reviewer's diligence.

The question for etc: what concretely forces the answer? Two shapes were available. (a) Prose reasoning — /architect's instructions say "consider quality at each layer", and the model reasons accordingly. (b) Matrix walk — /architect enumerates the touched layers, then for each layer walks its rubric items one by one, producing an explicit answer or a reasoned N/A per item, and `layer_review.py check` verifies no item was left blank.

**Decision:** The matrix walk. For each touched layer, `/architect` walks that layer's rubric (from the registry, per ADR-001) and produces, per rubric item, EITHER an explicit answer OR a reasoned N/A (a non-empty justification). An unanswered item is not permitted — the walk is the forcing function. The result is a structured Layer Impact Analysis table in design.md, one row per rubric item with (item id, criterion, answer-or-N/A, severity). `layer_review.py check` parses that table and exits non-zero if any touched layer has a rubric item with no answer and no reasoned N/A.

Crucially, the architect (the human, interactively) produces each answer; `layer_review.py` never fabricates answers (per spec BR-014). The engine enumerates what must be answered (`detect`) and verifies nothing was skipped (`check`); the human makes each decision. A reasoned N/A satisfies the check — "read-only feature, no migration" is a valid, recorded decision; an empty cell is not.

**Consequences:** *Positive:* completeness is structural — the index consideration cannot be skipped because the data-access rubric has that cell and `check` fails if it is blank; the failure mode that shipped the prod bug is now mechanically prevented; the analysis is auditable (the table records every decision, including reasoned N/As, so a reviewer sees what was considered and dismissed); reasoned N/A keeps the gate honest without forcing busywork (irrelevant criteria are explicitly dismissed, not silently skipped). *Negative:* friction — full breadth means the architect answers ~5-7 items per touched layer, which is real work (mitigation: reasoned N/A is fast for inapplicable items; advisory-default gating per ADR-003 means the build does not hard-block on incompleteness except in mandatory mode); a determined architect can produce low-quality answers to satisfy the cell (the walk forces the consideration to be MADE, not that it be made WELL — the quality of the answer is a human-judgment matter the gate cannot enforce); table-format coupling — `check` must parse the Layer Impact Analysis table, so the table format is a contract (mitigation: the format is defined once in the standards doc and asserted by tests).

**Alternatives considered:**

*Prose reasoning* (rejected). /architect's instructions say "reason about quality at each architectural layer." *Positive:* zero friction; no table to maintain; reads naturally. *Negative:* this is precisely what failed — "consider performance" is skippable prose, and the prod bug proves a diligent team skips it under pressure; nothing is auditable (no record of what was considered); `/build` has nothing to check (completeness is unobservable); regresses to the pre-incident state with better intentions but the same structural gap. The research is unanimous that implicit quality consideration is the failure mode these methods exist to prevent.

*Engine-generated answers* (rejected). `layer_review.py` reads the design and auto-fills the rubric answers. *Positive:* no architect friction. *Negative:* defeats the forcing function entirely — the point is to force the HUMAN to make each decision (BR-014); an auto-filled rubric is a rubber stamp with extra steps; the engine cannot know whether the indexes were actually planned, only whether the design mentions them. The human must produce the answer; the engine only enumerates and verifies.
