# ADR-F024-001: Conditional system-overlay sections — gate at hook layer, not role-manifest layer

**Date:** 2026-05-22
**Status:** Accepted

**Context:** `hooks/inject-standards.sh` emits identical content to every subagent regardless of role or task properties. Three sections are role- or task-specific: `### Git Commit Discipline` (relevant only to developer roles that write production code in a shared git index), `### Stub-Marker Grep Contract for spec-enforcer` (relevant only to the spec-enforcer role), and `### User-Flow Completeness for User-Facing ACs` (relevant only when the active task carries a User-flow acceptance criterion). Today all three are always emitted — approximately 500–750 tokens of irrelevant context per non-target dispatch. F024 makes emission conditional. The question is WHERE to apply the gate: inside the hook, or inside per-role manifests.

**Decision:** Gate at the hook layer (`hooks/inject-standards.sh`). The hook is one file; the role manifests are 18. The hook already receives the SubagentStart JSON — `agent_type` and `cwd` are extracted at lines 13–14 and are available throughout hook execution. The hook already conditionally injects three trailing sections (Active Task, INVARIANTS.md, Known Antipatterns) via `[[ -f ]]` / `[[ -d ]]` guards; F024 extends that same conditional-emission pattern to the three role- and task-gated sections. This yields all necessary gating from a single edit to a single file. The gating predicates are: `### Git Commit Discipline` emitted iff `agent_type` ∈ `{backend-developer, frontend-developer, devops-engineer}` or role is unknown (safe default); `### Stub-Marker Grep Contract for spec-enforcer` emitted iff `agent_type == "spec-enforcer"` (suppressed for unknown); `### User-Flow Completeness for User-Facing ACs` emitted iff the first in-progress task YAML contains an AC matching the User-flow sentence pattern (`As ` … `, navigate from`), or task YAML is absent/malformed (safe default). Base sections (TDD, Code Standards, Architectural Rules, Process, User Interaction, Research Discipline, Completion Discipline, Diagnostic Discipline, Sandbox Discipline) are invariant — never suppressed.

**Consequences:** *Positive:* One file to maintain instead of 18. Gating logic is co-located with content emission — the same file that writes a section also decides whether to write it, so the relationship is always visible and auditable by grepping a single path. Extends a pattern already present in the hook, so the idiom is familiar to any contributor who has read the existing trailing-injection block. Backward-compatible: invokers that supply no `agent_type` receive the safe-default behavior (all conditional sections emitted) matching today's behavior exactly for Git Commit Discipline; Stub-Marker Grep Contract is the one asymmetric exception where unknown suppresses rather than emits (the section is verbose and clearly role-specific; over-injection costs more here than under-injection). *Negative:* The hook accumulates more responsibilities over time; the gating logic is tightly coupled to the hook contract and must be updated when new developer roles are added to the allow-list.

**Alternatives considered:**

- **Gate via per-role manifest variants** (e.g., `agents/backend-developer-with-git.md`, `agents/backend-developer-without-git.md`, or a `no-git-discipline` variant for non-developer roles): Role manifests are loaded by the Claude Code runtime BEFORE the hook fires. Gating them would require either file-rewrite-at-dispatch-time (brittle, race-prone) or a symlink-swap mechanism (does not compose with the F020 profile-aware dispatch idiom). Beyond the mechanism problem, this approach explodes manifest count: each conditional section that varies by role multiplies the number of manifest files. Three conditional sections across 18 roles could produce dozens of manifest variants, fragmenting role identity and making it impossible to reason about what a "backend-developer" is without tracing through a version matrix. Rejected: wrong layer, wrong maintenance economics, incompatible with existing runtime loading order.
