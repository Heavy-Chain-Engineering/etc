# ADR-Ftmp-19e49f7c-001: Dispatch assembler cites design.md via requires_reading; never inlines its body

**Date:** 2026-05-23
**Status:** Accepted

**Context:** `scripts/dispatch_prompt.py` (F-2026-05-23) mechanizes the layer-3 per-dispatch prompt that `/build` Step 6a previously hand-authored. One construction choice is load-bearing: when the feature directory contains a `design.md` alongside `spec.md` (i.e., the feature went through `/architect`), the assembler must decide whether to embed design.md's body into the dispatch prompt or cite its path via `requires_reading` and let the subagent pull it via Read on demand.

The harness carries two prose anchors that disagree on this point. `skills/build/SKILL.md` lines 1091-1102 (F006 BR-005, authored before the F022-F024 subagent-context discipline crystallized) states the orchestrator "MUST additionally embed design.md's contents alongside spec.md in the briefing prompt — clearly delimited so the subagent can tell which artifact is which (intent vs. architecture)." `standards/process/subagent-dispatch.md` §Anti-patterns line 65 lists "Inlining design.md content" as anti-pattern #2: "If the dispatch quotes design.md sections, the subagent ends up reading the same content twice (once in the dispatch, once when it Reads design.md). Cite the section path; do not inline." The same standards doc §Forbidden in dispatch prompts line 55 directs orchestrators to replace inlined architectural constraints with "one line: 'See `design.md` §<Technical Constraints> for framework-version pins and security boundaries.'"

The standards doc post-dates the SKILL.md prose and is informed by the 2026-05-22 dispatch-examples retrospective at `.etc_sdlc/incidents/2026-05-22-dispatch-examples/`. Both captured ground-truth dispatches in that retrospective use cite-only — the observed best practice already matches the standards doc, not the older SKILL.md paragraph. The assembler operationalizes the standards doc; the SKILL.md prose is stale on this point and updated in sibling task 002.

Failure modes are asymmetric. Inlining design.md doubles the per-dispatch token count for any feature that went through `/architect` (design.md bodies in current /architect output run 800-2500 tokens) and produces redundant reads when the subagent's `requires_reading` already lists the path. Citing without inlining leaves the subagent to invoke Read once if and when the work requires the architectural detail — the agent retains control over WHEN to load it, and the dispatch prompt stays inside the 1,000-token target documented in `standards/process/subagent-dispatch.md` §Token budget.

**Decision:** The assembler emits `design.md`'s path into the `requires_reading` section of the dispatch prompt. It does not read design.md's body and does not splice any portion of design.md's content into the assembled prompt. The subagent invokes Read against the cited path on demand, exactly once, at the moment the architectural detail is needed.

This applies to every feature directory the assembler processes, whether or not `design.md` exists. When design.md is present, its path is added to the `requires_reading` list (sourced from task YAML's `requires_reading` field per BR-004 and GA-007 — task YAML's curation already includes design.md when /decompose deemed it relevant). When design.md is absent, the assembler emits nothing extra; the legacy "spec.md alone" shape is preserved by the absence of the design.md entry in task YAML.

The decision is the resolution of GA-001 in `gray-areas-spec.md`: cite-only via requires_reading, citing standards/process/subagent-dispatch.md §Anti-patterns (lines 61-69) and §Forbidden in dispatch prompts (lines 51-59) as the load-bearing authority. The 2026-05-22 dispatch examples in `.etc_sdlc/incidents/2026-05-22-dispatch-examples/` are the empirical confirmation that observed best practice has already converged on cite-only.

**Consequences:**

*Easier.* Subagents read design.md at most once per dispatch (the Read invocation triggered by the requires_reading path), not twice (once via the inlined dispatch body, once via the requires_reading path the agent follows). Per-dispatch token count drops by the size of design.md's body — meaningfully, since /architect output runs 800-2500 tokens. The 1,000-token dispatch target in `standards/process/subagent-dispatch.md` §Token budget becomes achievable for features that went through /architect (currently the dispatch baseline for those is ~1,400 tokens; cite-only with concurrent F024 reductions targets ~800). The subagent retains control over WHEN to load design.md — for tasks that do not need the architectural detail, it can skip the Read entirely, which is exactly the behavior the standards doc's "smallest sufficient context" anchor calls for.

*Harder.* Subagents lose the synchronous-load convenience of having design.md already inlined in the prompt context. Any task whose first action depends on an architectural constraint must invoke Read against the cited path before proceeding — one additional tool call relative to the inline path. For very short design.md bodies (under ~200 tokens) this is a marginal loss; for longer ones it is a strict win on token economics, which is the dominant case post-/architect.

*Deferred.* An operator-facing toggle (e.g., `--inline-design-md` flag on `dispatch_prompt.py assemble`) to force inlining for dispatches that genuinely need synchronous design.md context — for example, dispatches where design.md is under 200 tokens and skipping the Read tool call saves a wall-clock turn. Deferred until measured demand surfaces in a build retrospective; no current incident motivates the toggle, and adding it preemptively expands the assembler's surface against the GA-005 "embed in code, cite the standards doc" simplicity stance.

*Cannot defer.* `skills/build/SKILL.md` lines 1091-1102 must be updated in sibling task 002 (Wave 1, post-this-task) to remove the "embed design.md alongside spec.md" prose and replace it with cite-only semantics consistent with the standards doc and the assembler's behavior. Without that update, the SKILL.md prose remains in direct contradiction with the assembler's emitted output — an operator reading SKILL.md and observing the assembler's behavior would see the harness contradict itself. The update is mechanical (replace the embed paragraph with a one-line pointer to `scripts/dispatch_prompt.py` and `standards/process/subagent-dispatch.md`) and is in scope for task 002 per BR-011.
