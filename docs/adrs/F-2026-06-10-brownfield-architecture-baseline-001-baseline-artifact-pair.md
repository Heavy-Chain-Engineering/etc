# ADR F-2026-06-10-brownfield-architecture-baseline-001: Baseline Artifact Pair — Machine YAML Source + Generated ARCHITECTURE.md Twin

**Date:** 2026-06-11
**Status:** Accepted
**Context:** The ratified architecture baseline needs both machine consumers (gates, checker generation, rule-and-sweep) and human/agent readers (tier-0 context). The covr forensics showed docs/-path conventions go undiscovered, and hand-maintained doc pairs drift. profiles.lock is ADR-fixed as plain text and cannot carry rich state (F020-005); the rich-YAML-under-.etc_sdlc precedent is profiles.yaml; contract-completeness ADR-002 fixes the schema_version forward-compat contract.
**Decision:** Two artifacts per repo: (1) `.etc_sdlc/architecture-baseline.yaml` — the single editable-by-tooling source of truth (schema_version: 1, closed status enum, claim ledger, exemplar registry, rules, seams, auditable confidence), owned exclusively by `scripts/baseline.py`, atomic writes, warn-and-skip on future schema_version; (2) `ARCHITECTURE.md` at the repo ROOT — a generated render of the machine file, joining DOMAIN.md/PROJECT.md as the third root tier-0 artifact, regenerated at (re-)ratification, carrying a "generated — edit via /init-project --phase=baseline" header. Root placement over docs/ because discoverability failures, not prose quality, were the expensive covr failure.
**Consequences:** Easier: no two-way sync or drift between human and machine views; gates read one file; agents discover the doc at root. Harder: humans cannot hand-polish ARCHITECTURE.md (edits are overwritten); render quality bounded by the renderer. check-tier-0.sh gains ARCHITECTURE.md in its self-exemption list only — never in its block condition (forward-only).
