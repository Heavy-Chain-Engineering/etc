# ADR-F020-006: etc adopts community-canonical style guides; never authors its own

**Date:** 2026-05-16
**Status:** Accepted
**Context:** Each per-profile bindings file needs an authoritative source. Two options: (a) etc authors its own per-language style guides (the "we know better" stance); (b) etc adopts community-canonical sources and cites them (Google styleguide, Airbnb, Mozilla, PEP 8, Rust API Guidelines, Apple Swift API Design Guidelines). Phase 2 web research surfaced [Kristories/awesome-guidelines](https://github.com/Kristories/awesome-guidelines) — a curated meta-repo covering 40+ languages with abundant existing authority. Choice (a) duplicates work that's already done well by communities with deeper context than etc's authors have for each language.
**Decision:** Adopt and cite. Every `standards/code/profiles/<profile>/*-bindings.md` MUST contain a `## Source` section naming at least one external URL or document. The python profile cites Google Python Style Guide + PEP 8 + ruff rule docs. New profiles cite per Section "Best Practices" of `docs/audits/F020-language-coupling-audit.md`. A test enforces the `## Source` section presence per AC-009.
**Consequences:** *Positive:* zero etc-authored bikeshedding; partners see their own community's canon enforced (familiar); the per-profile work is mechanical (cite + bind tooling). *Negative:* if a profile's community canon is contested (multiple competing style guides — JavaScript with Airbnb vs StandardJS, for example), the per-profile spec must pick one and document why; the rationale lives in the per-profile spec, not in etc's voice.
