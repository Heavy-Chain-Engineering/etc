# ADR-F020-001: Profile as a first-class harness primitive

**Date:** 2026-05-16
**Status:** Accepted
**Context:** etc's enforcement gates (5 hooks), code standards (14 files), agent manifests (3 manifests), and skill bodies (2 skills) hardcode Python tooling (`pytest`, `mypy`, `ruff`, `uv`). The audit at `docs/audits/F022-language-coupling-audit.md` inventories 24 surfaces. Closing the gap requires a unifying abstraction. The choice is between (a) ad-hoc per-tool branching inside each hook, (b) a config-driven "language detection" feature that's still implicit, or (c) a first-class profile primitive that names the abstraction explicitly.
**Decision:** Introduce a profile as a first-class harness primitive. A profile is a directory at `standards/code/profiles/<profile>/` that declares (i) detection markers, (ii) per-rule tool bindings, (iii) gate scripts. Every supported language, framework, format, or platform is represented by a profile. Profiles MUST NOT be implicit or inferred — explicit directory presence is the contract.
**Consequences:** *Positive:* uniform pattern across languages; per-profile content is self-contained and grep-friendly; adding a new language is mechanical (copy reference profile, swap bindings). *Negative:* one more abstraction to learn; profile directory structure becomes load-bearing — future restructuring is expensive once profiles exist downstream.
