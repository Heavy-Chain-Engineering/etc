# ADR-F025-002: `shipped/` is a state, not a one-way door — extensions reopen the feature dir in place

**Date:** 2026-05-22
**Status:** Accepted

**Context:** F025 ships `/build --extend "<problem>"` as the refinement lane for shipped features. Pre-F025, the F009 lifecycle treats `.etc_sdlc/features/shipped/` as terminal: once `/build` Step 7c.1 moves a feature directory from `active/` to `shipped/` and writes the `etc/feature/F<NNN>/release` tag, the directory is audit-frozen and no further work runs against it. Real engagements have a long tail of post-ship refinement — wrong component library, missing ETL step, stakeholder-surfaced UI nits — that today exits the harness and loses every governance gate within hours (TDD, F021 diagnostic-evidence, F020 verify-green, structured planning). The question for F025 is where extension work physically lives and how its audit trail attaches to the original feature. Three candidates: (a) reopen — move the feature dir back to `active/` for the extension duration, then forward again to `shipped/` on re-close; (b) append-only sub-feature directory under the shipped feature (`features/shipped/F042-<slug>/extensions/<extend_id>/`); (c) allocate a new F-ID per extension (`F042`, `F042.1`, `F042.2`, …).

Option (b) preserves the "shipped is terminal" invariant but fragments the audit trail across N sibling directories. A reader investigating F042 must now consult the root dir AND every `extensions/<extend_id>/` subdir to reconstruct what shipped. `/metrics` rollups that read `state.yaml` per feature would need to walk the extensions tree, recombine ACs, and reason about which release tag is current. The `spec.md`, `design.md`, ADR index, and `value-hypothesis.yaml` either get duplicated per extension (drift risk) or split across dirs (orientation cost). Subagents dispatched into an extension need the original context-pack; with sub-feature dirs they need a context-pack assembler that walks both the root and the current extension dir.

Option (c) keeps each extension as a first-class feature but pollutes the F-ID space and breaks the F-ID-as-feature semantic that the rest of the harness depends on. `F042.1` is not an independent feature — it has no independent `spec.md`, no independent `value-hypothesis.yaml`, no independent architect handoff. Tags become ambiguous (`etc/feature/F042.1/release` looks like a feature release but represents a refinement of F042). Operator memory has to track which F-IDs are roots and which are extensions; `/metrics` rollups have to special-case the dotted IDs; F023's `allocate-next` POSIX-atomic counter has to learn an entirely new ID shape. The semantic — one F-ID = one shipped product increment with one spec — is load-bearing for `/spec` classification, the spec-enforcer adversarial review, and the value-hypothesis contract. Diluting it for refinements is a small local convenience with a structural cost everywhere else.

Option (a) treats `shipped/` as one state in a two-state lifecycle rather than a terminal sink. The feature dir moves shipped → active on `--extend`, the extension augments the SAME dir in place (`state.yaml.extends` appends; `release-notes.md` gains an `## Extensions` section; original `spec.md`, `design.md`, ADRs untouched), and the dir moves active → shipped again on re-close. The audit trail accretes in one place. Reads stay local — one dir, one `state.yaml`, one release-notes file. The F-ID is preserved as the feature's identity; extension versioning happens at the release-tag suffix (`etc/feature/F042/release_<extend_id>`), which is append-only per F021 BR-008. F022's `shutil.move` fallback handles the dir moves; F023's POSIX-atomic allocator is unaffected (no new F-IDs allocated). Subagents dispatched into the reopened feature read the SAME context-pack that was assembled at original ship time, plus the new `<problem>` statement.

**Decision:** `shipped/` is a state, not a one-way door. `/build --extend` moves the target feature directory from `.etc_sdlc/features/shipped/F<NNN>-<slug>/` back to `.etc_sdlc/features/active/F<NNN>-<slug>/` for the duration of the extension. The original `spec.md`, `design.md`, ADRs, `value-hypothesis.yaml`, `verification.md`, and `release-notes.md` are NOT replaced — they remain in place, and the extension augments via append-only `state.yaml.extends[]` and an append-only `## Extensions` section in `release-notes.md`. On successful re-close, the dir moves forward to `shipped/` again and a versioned release tag (`etc/feature/F<NNN>/release_<extend_id>`) is written alongside the original `etc/feature/F<NNN>/release` (which is preserved byte-equivalent). The F-ID is invariant across the lifecycle; extension versioning lives only in the release-tag suffix and in `state.yaml.extends[N].extend_id`. Endpoint discipline: a reopened extension MUST eventually re-close (`/metrics` surfaces in-flight extensions as unclosed).

**Consequences:** *Positive:* one feature, one directory, one append-only audit trail — no fragmentation; `/metrics` rollups read a single `state.yaml` per feature; subagents inherit the original context-pack without an assembler step; the F-ID-as-feature semantic stays intact (no dotted IDs, no pollution of `allocate-next`); composes cleanly on F021 (append-only tags), F022 (`shutil.move` for both directions), F023 (no new ID allocation), and F024 (conditional onboarding inherited by extend dispatches); release-tag versioning at the suffix is a natural extension of F021 BR-008 rather than a new tag convention. *Negative:* "shipped" is no longer terminal — a convention shift operators must internalize (mitigated by `standards/process/build-extend.md` per BR-011 and by `/metrics` surfacing in-flight extensions); the same feature dir alternates between `active/` and `shipped/`, so a stale `ls` snapshot may not reflect the current state (mitigated by `state.yaml.extends[N].completed_at` being the source of truth); concurrent `/build --extend` invocations on the same feature from different machines have a first-mover wins on the `shutil.move` and the second-mover gets `shutil.Error` (documented as expected behavior per EC-004, matches F022's three-branch failure shape).

**Alternatives considered:** Append-only sub-feature directory at `features/shipped/F042-<slug>/extensions/<extend_id>/` (rejected — fragments the audit trail across N sibling dirs; forces `/metrics` rollups to walk the extensions tree and recombine; duplicates or splits the context-pack that subagents must read; turns one-feature-one-dir into one-feature-N-dirs with no offsetting benefit). New F-ID per extension, e.g., `F042 + F042.1 + F042.2` (rejected — pollutes the F-ID space with non-features; breaks the F-ID-as-feature semantic that `/spec`, spec-enforcer, value-hypothesis, and `/metrics` all depend on; requires `allocate-next` to learn a new ID shape; ambiguates tags — `etc/feature/F042.1/release` looks like a feature release but is a refinement; operator memory now distinguishes root IDs from extension IDs).
