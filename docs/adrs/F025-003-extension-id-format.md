# ADR-F025-003: Extension ID format — 8-char hex from `time.time_ns()` + `os.urandom(2)` + sha256 truncate

**Date:** 2026-05-22
**Status:** Accepted

**Context:** F025 introduces `/build --extend "<problem>"` — a refinement lane for shipped features. Each successful extend cuts a new release tag (`etc/feature/F<NNN>/release_<extend_id>`), appends an entry to `state.yaml.extends`, writes a `release-notes.md` sub-section, and emits an `extend_dispatch` audit-log row. All four surfaces reference the same `extend_id`. The identifier needs to: (a) be operationally collision-free across machines without a central registry (the same race class F023's POSIX-atomic `allocate-next` was built to solve for sequential `F<NNN>` allocation), (b) sort lexicographically by creation time so `/metrics` rollups and `git tag --list` output present extends in chronological order without a schema-aware sort, (c) fit human-scannable git tag paths and directory suffixes (rhyming with F023's `Ftmp-<8-hex>` shape so the format reads as a single family), and (d) be generable from Python stdlib only — F025 inherits the zero-new-deps constraint that bounds F021/F022/F023.

**Decision:** Extension IDs take the form of an 8-character lowercase hex string generated by `scripts/extend_resolver.py::generate_extend_id()`:

```python
def generate_extend_id() -> str:
    timestamp_ms = time.time_ns() // 1_000_000   # 6 bytes
    randomness = os.urandom(2)                    # 2 bytes
    raw = timestamp_ms.to_bytes(6, "big") + randomness
    return hashlib.sha256(raw).hexdigest()[:8]
```

Example: `01b5a3c7`. The construction is a stdlib UUID7-like shape: a millisecond timestamp prefix supplies time-ordering, 2 bytes of `os.urandom` supply intra-millisecond randomness, and sha256-truncate-to-8-hex compresses the 8-byte payload into a directory/tag-safe token. The output is 8 hex characters — identical in width to F023's `Ftmp-<8-hex>` so the two ID families share a visual shape and one regex (`^[0-9a-f]{8}$`) covers both. The 8-character width is deliberate: long enough to be operationally collision-free at F025's invocation rate (one extend per shipped feature per day is already aggressive), short enough to keep `etc/feature/F042/release_01b5a3c7` legible in `git tag --list` output and `state.yaml` diffs.

**Consequences:** *Positive:* IDs sort chronologically by string comparison — `/metrics` and `git tag --list 'etc/feature/F042/*'` present extends in the order they were created, without a schema-aware parser. Cross-machine generation is collision-free in practice (the timestamp + 2 bytes urandom payload would require two extends on the same feature inside the same millisecond to collide on the sha256 prefix, then collide again on the 8-hex truncation — operationally never). Zero new dependencies — `time`, `os`, `hashlib` are all Python 3.11+ stdlib. The 8-hex shape rhymes with F023's `Ftmp-<8-hex>`, so audit tooling, `grep`, and operator muscle memory carry over from temp IDs to extension IDs. *Negative:* The construction is not a standards-conformant UUID7 — third-party tooling that expects RFC-style UUIDs cannot parse it. F025 does not export the ID to any external system, so this is a closed-world cost. The sha256 truncate trades a small amount of timestamp precision for a uniform hex shape; two extends in the same millisecond on the same feature MAY produce non-monotonic 8-hex strings (the timestamp prefix is hashed, not preserved verbatim). The collision space is approximately 50 days of timestamp values before two extends in the same millisecond on the same feature would meaningfully race — operationally unreachable. The trade is accepted because the visual-shape match with F023 and the stdlib-only construction outweigh strict monotonicity at sub-millisecond resolution.

**Alternatives considered:**

- **Pure UUID4 (`uuid.uuid4().hex[:8]`):** Stdlib, collision-free at 8-hex width via random truncation, but **not time-ordered**. `git tag --list 'etc/feature/F042/*'` would present extends in lexical order unrelated to creation time; `/metrics` rollups would need to join through `state.yaml.extends[].started_at` to recover chronology. The chronological-sort property is load-bearing — operators reading audit output expect oldest-first or newest-first ordering without an extra join. Rejected.

- **Sequential `/N` suffix (`etc/feature/F042/release/1`, `/2`, `/3`):** Re-introduces the exact allocate-next collision class across machines that F023 was built to solve for `F<NNN>` sequential IDs. Two operators running `/build --extend` on the same shipped feature from different machines (EC-004) would both read the highest existing `/N`, increment, and write — producing two tags at the same `/N` value or one operator silently overwriting the other's release notes entry. Solving this would require either re-using F023's POSIX-atomic `allocate-next` (adding a second allocation race surface to maintain) or a central registry (violates F025's single-process local-filesystem deployment posture). Rejected on the principle that F025 should not re-introduce a collision class a prior feature was built to eliminate.

- **`uuid7` PyPI dependency:** A proper UUID7 generator is available as multiple PyPI packages (`uuid7`, `uuid6`, `uuid-utils`). They produce standards-conformant time-ordered UUIDs and would technically improve sub-millisecond monotonicity. Rejected on the no-new-deps policy that bounds F021's `scripts/diagnostic_evidence.py`, F022's `shutil.move` fallback, and F023's `scripts/feature_id.py`. The stdlib construction (`time.time_ns()` + `os.urandom(2)` + `hashlib.sha256`) reproduces the time-ordering property of UUID7 at the operational fidelity F025 needs (millisecond resolution, collision-free across machines, 8-hex width). A new dependency for a property the stdlib already supplies is unjustified weight on the install path.
