# ADR F-2026-06-10-brownfield-architecture-baseline-003: Verify Claims, Never Honor Docs — the Claim Ledger and the Parallel Engine

**Date:** 2026-06-11
**Status:** Accepted
**Context:** Teams with weak process have weak docs — stale, aspirational, or wrong — and etc's own generated DOMAIN.md at covr contained a factually wrong system-boundary claim. Honoring discovered docs uncritically swaps one wrong context for another (the fake-client-fidelity failure family). Meanwhile the verification work is embarrassingly parallel, and the 2026-06-10 covr forensic workflow (7 parallel read-only agents, structured-output schemas, commit-level evidence) validated the execution shape.
**Decision:** Every load-bearing claim in a discovered artifact — including etc's own prior tier-0 artifacts on re-init — is checked against the tree and classified into the closed enum VERIFIED | STALE | ASPIRATIONAL | CONTRADICTED with file-level evidence, recorded in the baseline's claim ledger. Only VERIFIED claims enter agent context silently; every other classification surfaces at ratification through the matrix-walk forcing function (layered-review ADR-002 shape: the engine enumerates, the human resolves every cell, the engine never fabricates). DISCOVER and VERIFY execute as a parallel read-only agent fan-out (per-artifact, per-concern, per-repo) with bounded sampling on giant repos — bounds reported, never silent.
**Consequences:** Easier: tier-0 context becomes trustworthy; a hostile or stale doc costs one ratification decision instead of a misbuilt feature; wall-time stays inside the 15-minute budget. Harder: init is slower and noisier on doc-heavy repos; bounded sampling means VERIFY is evidence-based, not exhaustive. The claim ledger stays baseline-specific (no generalized doc-verification framework) until a second consumer exists — YAGNI.
