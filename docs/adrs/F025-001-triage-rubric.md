# ADR-F025-001: Triage rubric is rule-based and deterministic, not LLM-classified

**Date:** 2026-05-22
**Status:** Accepted

**Context:** `/build --extend "<problem>"` (F025) routes a refinement request into one of three lanes — Light (auto-dispatch, no spec), Medium (micro-spec then dispatch), Heavy (refuse + redirect to `/spec`). The classifier sits on the hot path: every extend invocation runs it before any state change. Two structural choices were available for the classifier itself. (a) Rule-based deterministic — a small Python function on top of regex predicates over the operator's `<problem>` text and the target feature's context-pack: file-path mention count, architectural-keyword scan, ADR-amendment language detection. (b) LLM-based — pass the problem string + context-pack into a small model, ask for a Light/Medium/Heavy label.

Triage failure modes are asymmetric. A misclassified Heavy as Light or Medium silently expands a shipped feature with architectural-impact work — exactly the post-ship scope-creep class F025 exists to defend against. A misclassified Light as Medium adds 2-3 spec questions the operator did not need. A misclassified Medium as Heavy refuses work that should have proceeded; the operator sees the refusal message and either re-invokes with `--triage medium` or files a fresh `/spec`. The Heavy-as-Light failure is the only outcome that defeats F025's value hypothesis; the other two are recoverable on the spot.

The classifier is invoked unattended (the operator types the `/build --extend` command and walks away while the dispatch runs). Whatever the classifier emits is what the harness acts on, modulo the explicit `--triage` override flag. Confidence in classifier behavior must therefore be precommit, not posthoc.

**Decision:** Rule-based deterministic triage. `scripts/extend_resolver.py::classify(problem, target_dir) -> Literal['light', 'medium', 'heavy']` is a pure function over stdlib `re` matches on the problem text plus filesystem reads of the target's context-pack. The gating predicates are documented verbatim in `standards/process/build-extend.md` and spec BR-002:

- **Light** ⇔ problem text names ≤3 specific file paths (regex match on `.py|.ts|.tsx|.md|.sh|.yaml|.yml` suffixes) AND contains no architectural keywords.
- **Heavy** ⇔ problem text contains ≥1 architectural keyword (`redesign`, `rearchitect`, `swap framework`, `migrate to`, `replace with`, `restructure`) OR explicit ADR-amendment language (`amend ADR`, `revise ADR`, `change the architecture`).
- **Medium** ⇔ everything else.

The operator overrides the auto-classification with an explicit `--triage light|medium|heavy` flag. The override is the documented escape hatch for novel phrasings the rubric misses; it appears in the audit log alongside the auto-classified outcome so subsequent reviewers see both the rubric's verdict and the operator's correction.

No model call; no PyPI dependency; no network. Stdlib `re` and `pathlib` only.

**Consequences:** *Positive:* deterministic — the same `<problem>` string against the same context-pack returns the same label across machines, across days, across operator sessions; auditable — the predicates are eight lines of regex visible in the helper script and reproduced verbatim in the standards doc, so an operator reviewing a past extend can rerun the classifier in their head; zero marginal cost — no token spend per invocation, no rate-limit pressure, no model-availability dependency on the hot path; fully testable — `tests/test_extend_classify_rubric.py` runs 12+ parametrized cases in milliseconds with no mocks; no new dependency surface — the F025 pyproject.toml diff is empty. *Negative:* rubric brittleness on novel phrasings — an operator who phrases an architectural change without any of the six keywords (e.g., "make the auth different") gets a Medium classification rather than Heavy; the `--triage` override is the mitigation but requires the operator to recognize the misclassification themselves; the keyword list is a small surface that requires occasional expansion as language drifts (additions are append-only standards-doc edits, not code changes); an adversarial operator who deliberately phrases a Heavy problem as Light to bypass the refusal can do so — documented as an explicit non-defense in design.md (the operator works against their own audit trail).

**Alternatives considered:**

*LLM-based classification* (rejected). Pass the `<problem>` text plus the target feature's context-pack into a small model and ask for a `light | medium | heavy` label. *Positive:* handles novel phrasings the keyword rubric misses; degrades gracefully on language drift; reads naturally to operators who already use LLM-mediated tools elsewhere. *Negative:* non-determinism — the same problem string against the same context-pack may emit different labels across invocations, defeating audit reproducibility; token cost on every extend dispatch on the hot path; new external dependency (model availability + API surface + rate limits) gating a control-plane decision; semantic testing requires either golden-output fixtures that drift with model versions or live model calls in CI, both of which the F021 diagnostic-discipline standard discourages; the failure mode that matters (Heavy misclassified as Light) is exactly the case where an LLM's confident-sounding wrong answer is harder to catch than a rubric's literal regex miss. The asymmetric cost of misclassification favors the predictable-and-wrong-on-edges classifier over the unpredictable-and-occasionally-right-on-edges one.

*Trust operator's choice unconditionally* (rejected). Drop the classifier entirely; the operator passes `--triage light|medium|heavy` on every invocation and the harness dispatches accordingly. *Positive:* simplest possible implementation; no rubric to maintain; no false-positive Heavy refusals; respects operator agency. *Negative:* loses the Heavy-refusal safety property that anchors F025's value hypothesis — the scope-creep guard exists precisely because operators in the moment of "let me just fix this one thing" routinely misjudge the architectural reach of their own requests; removes the default-deny posture for novel phrasings (the rubric refuses Heavy unless told otherwise; the trust-operator design proceeds unless told otherwise); produces a worse audit trail (every dispatch records the operator's self-assessment with no independent classification to compare against, so retrospective review cannot distinguish "rubric agreed" from "operator overrode rubric"); inverts the F025 design intent from "harness defends post-ship scope" to "harness executes whatever the operator typed." The rubric is the defense; without it, F025 reduces to a shell around `/build` that skips Step 1.

The chosen design keeps the rubric as the default-deny gate and the `--triage` flag as the escape hatch — the operator can override but must do so explicitly, leaving a trace in the audit log that subsequent reviewers can inspect.
