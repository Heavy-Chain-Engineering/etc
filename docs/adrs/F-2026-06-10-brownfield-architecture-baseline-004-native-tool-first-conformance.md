# ADR F-2026-06-10-brownfield-architecture-baseline-004: Native-Tool-First Conformance, F020 Profile Fallback

**Date:** 2026-06-11
**Status:** Accepted
**Context:** Covr already had @nx/enforce-module-boundaries and the team hand-built check-pbj-patterns.mjs within 48 hours of seeing violations — the tooling layer exists everywhere; the missing input is ratified rules. Industry prior art (architecture fitness functions: ArchUnit, dependency-cruiser, ESLint boundaries) is mature. The harness already has a proven per-profile gate-delivery mechanism (F020) and a proven JSON dispatcher contract (runtime-verify ADR-001), including the staged-rollout precedent (runtime-verify.sh exists only in the python profile; others warn-and-skip).
**Decision:** Checker generation prefers configuring the project's NATIVE fitness-function tool from the ratified rules (rules survive after etc leaves). Rules a native tool cannot express fall back per-rule to a generated checker delivered as `standards/code/profiles/<p>/baseline-verify.sh`, dispatched by `hooks/baseline-verify.sh` — a clone of runtime-verify.sh (conductor-invoked, never a Stop hook; JSON stdin; `{"results":[{rule_id,status:pass|fail|no-check,evidence}]}` stdout; dispatcher always exits 0; timeout → synthetic fails; unknown statuses aggregate as fail). Each rule records `enforced_by: native|generated|human-judgment` and a `mechanizable` flag (layered-review ADR-004 graduation metadata). v1 ships the python reference profile only. Rule-and-sweep appends rules to a ratified baseline without reopening ratification (rules accrete; the architecture target does not silently change).
**Consequences:** Easier: teams keep their toolchain; etc ships less code; conformance verdicts are structured and per-rule. Harder: native-config generators accrete per ecosystem over time; the per-rule enforced_by split adds bookkeeping. The conductor (not the dispatcher) owns gate semantics, exactly as /build consumes runtime-verify.
