# ADR-Ftmp-19e49f7c-002: Dispatch section ordering is embedded in assembler source, not parsed from the standards doc

**Date:** 2026-05-23
**Status:** Accepted

**Context:** `scripts/dispatch_prompt.py` (F-2026-05-23) mechanizes the eight required dispatch-prompt sections enumerated at `standards/process/subagent-dispatch.md` §Required dispatch sections (lines 27-47): feature intent, task intent, task identifier, required reading, files in scope, acceptance criteria, cross-task awareness, report-back format. The assembler needs that ordering as data on every invocation. Two structural choices were available. (a) Parse the standards doc at runtime — open `standards/process/subagent-dispatch.md`, walk the `§Required dispatch sections` heading, extract the numbered list items, derive section identifiers from the bold labels. (b) Embed the ordering as a module-level constant in `scripts/dispatch_prompt.py` with a comment citing the standards doc as the canonical source, and rely on PR-time review to keep the two in sync when either side changes.

The dispatch-prompt assembler sits on the `/build` Step 6a hot path. Every task dispatch in every feature build invokes it. Whatever ordering the assembler emits is what every subagent across every wave receives; a mid-build markdown-parse failure would block the wave and require a manual escape. The standards doc itself is human-authored markdown — heading levels, prose interleaving between list items, and future stylistic edits (e.g., adding a sub-bullet under one of the eight sections, or reformatting the list as a table) are not stable inputs for a runtime parser. The eight-section list has been the same shape through F022 (intent-gap observation), F024 (system-overlay split), and F025 (extend triage). The drift cost of "code says one thing, standards doc says another" is detectable at review time when either file changes; the drift cost of a runtime markdown parser silently mis-identifying section boundaries is not detectable until a dispatch is observably wrong.

F020 established the harness precedent for this trade. Per ADR-F020-002 and the `standards/code/profiles/python/` layout, language-specific vocabulary (ruff codes, mypy flags, pytest invocations) lives as data inside profile files while the rationale and concept ("function size limit," "never silently swallow errors") lives as prose in `standards/code/clean-code.md` and `standards/code/error-handling.md`. Agents read both: the rule for understanding, the binding for the literal token. The standards doc is the human-readable canonical statement; the profile vocabulary is the machine-readable mirror. The mechanism for keeping the two in sync is deliberate review at PR time when either file moves, not runtime parsing of one from the other.

**Decision:** Embed the ordering in code; cite the standards doc.

`scripts/dispatch_prompt.py` declares a single module-level constant `SECTION_ORDER` near the top of the file, after imports and before any function definitions. The constant is an ordered tuple of eight section identifiers — one entry per required section in the order specified by the standards doc:

```python
# Section ordering for the assembled dispatch prompt.
# Canonical source: standards/process/subagent-dispatch.md §Required dispatch sections.
# When the standards doc changes the section ordering, update this constant in
# the same PR that edits the doc; the two MUST move together.
SECTION_ORDER = (
    "feature_intent",       # 1. one paragraph from spec.md Summary
    "task_intent",          # 2. optional; omitted when task YAML lacks intent:
    "task_identifier",      # 3. full path to task YAML
    "required_reading",     # 4. paths with <=8-word commentary
    "files_in_scope",       # 5. one path per line
    "acceptance_criteria",  # 6. verbatim from task YAML
    "cross_task_awareness", # 7. wave-plan peers + file-set isolation
    "report_back_format",   # 8. standardized template
)
```

The comment immediately above the constant names `standards/process/subagent-dispatch.md §Required dispatch sections` as the canonical source. The constant is referenced from a single assembler function (the top-level orchestrator that iterates the tuple and dispatches to per-section formatters) — it MUST NOT be duplicated across multiple call sites in the script, because duplication defeats the cite-and-mirror discipline by hiding which definition is canonical inside the codebase.

When the standards doc changes the section ordering — adding a ninth section, renaming one, reordering two — the constant is updated in the same review-eligible commit that edits the doc. The reviewer's job is to confirm the two sides match; the constant is the load-bearing artifact for that confirmation.

**Consequences:** *Easier:* the assembler is deterministic — the same task YAML against the same spec.md emits the same prompt ordering across machines, across days, across operator sessions; no runtime markdown-parsing dependency, so the assembler is stdlib-only (composes on F022's stdlib-only helper convention named in the spec); no dependence on the standards doc's filesystem location, so the script remains relocatable inside `~/.claude/scripts/`; the eight section identifiers are grep-anchorable inside the codebase, so a future reader inspecting which sections the assembler emits gets the answer from one file rather than a doc-parse trace; standards-doc and code stay in sync via deliberate PR-time review when either changes, the same mechanism F020 established for the universal-rule / per-profile-binding split (ADR-F020-002).

*Harder:* a standards-doc change without a corresponding code change is silent — the assembler keeps emitting the old ordering, and the drift is only visible when an operator reads a dispatch and notices the structure no longer matches the doc. Mitigation: a small lint or test that walks the standards doc's `§Required dispatch sections` numbered list, extracts the bold labels, and compares the count + ordering to `SECTION_ORDER`. This is deferred to a follow-up because (i) the drift has not been observed across F022-F024-F025, (ii) the lint introduces the markdown-parsing fragility the present decision rejects on the hot path (acceptable in a CI lint, where a failure blocks merge rather than a dispatch), and (iii) the cost of the manual review discipline is low while the standards doc has a single canonical author.

*Deferred:* a runtime-parsing fallback for downstream projects that fork the standards doc layout — e.g., a customer who renames `§Required dispatch sections` to `§Dispatch sections (required)` and expects the assembler to follow. Not needed at v1 because etc owns the standards doc; the canonical statement does not have multiple authors. If a fork case emerges, the v2 design is a `--standards-doc <path>` flag that opts into a parsed fallback, with the embedded constant as the default.

*Cannot defer:* the constant MUST be a single module-level definition. Duplicating the ordering across two formatter functions, or rebuilding the tuple inside a class constructor, defeats the cite-and-mirror discipline by hiding which definition is canonical inside the codebase. A reviewer who edits the standards doc and grep'd for one copy of the constant would miss the second. The "single module-level definition" rule is the load-bearing constraint that makes the discipline visible.

**Alternatives considered:**

*Runtime markdown parsing of `standards/process/subagent-dispatch.md`* (rejected). Open the standards doc on every assembler invocation, locate `## Required dispatch sections`, walk its numbered list, extract each item's bold label, derive section identifiers from those labels. *Positive:* the standards doc becomes the literal single source of truth — any edit to the doc's section list is picked up by the next dispatch with no code change required; eliminates the manual sync discipline; reads naturally to operators who expect "the doc IS the spec." *Negative:* markdown parsing is fragile against the inputs the standards doc actually receives over time — heading-level changes (`##` becomes `###` under a parent section), prose interleaved between list items, future reformatting of the list into a table or a definition list, line-break changes inside list items, future additions of explanatory sub-bullets the parser would have to decide whether to count as section delimiters; a parse failure on the hot path blocks every wave's every dispatch until a fix lands, with the recovery path being either a manual edit of the standards doc to placate the parser or a code change to relax the parser, both of which are worse outcomes than the silent-drift case the embedded-constant design accepts; introduces a runtime filesystem dependency on the standards doc being present at a known path, which complicates the script's relocatability under `~/.claude/scripts/`; the failure mode that matters (assembler emits a wrong ordering on the hot path) is the case where a runtime markdown parser's silent-success-with-wrong-output is harder to catch than a code constant's grep-anchored, review-time visibility. The F020 profile pattern made the same trade for the same reasons: vocabulary in code, rationale in standards doc.

*Duplicate the ordering across each assembler function* (rejected). Rather than a single module-level constant, let each per-section formatter function know its own position by integer literal, or let a top-level assembler function inline the tuple at the call site, or let a `DispatchPrompt` class duplicate the ordering inside `__init__` for clarity. *Positive:* removes a layer of indirection — each function reads as self-contained. *Negative:* defeats the cite-and-mirror discipline — a reviewer who edits the standards doc and grep'd for "feature_intent" inside the script would find multiple definitions and have to reason about which is canonical; doubles the chance that a future edit updates one copy and not the other, producing a silent-drift outcome inside the codebase that is worse than the silent-drift outcome between code and standards doc (the latter has the standards doc as an independent check; the former does not); the discipline being protected by this ADR is "one constant, one comment, one cite" — duplication breaks the protected invariant.

*Generate the constant from the standards doc at build/install time* (rejected). A pre-commit hook or `install.sh` step parses the standards doc once at install time, writes the resulting tuple into a generated file (e.g., `scripts/_dispatch_sections_generated.py`), and the assembler imports from there. *Positive:* the standards doc remains the source of truth in a stronger sense than the embedded constant — the generation step is the manual sync discipline mechanized; runtime stays parser-free. *Negative:* introduces a generated file the assembler depends on, which means a fresh clone before install has a broken assembler; install-time parsing still has to handle the same heading-level/prose-interleaving fragility as runtime parsing, just earlier; the generated file becomes a new artifact requiring its own ".do not edit by hand" discipline and a way to detect drift between the generated file and the standards doc when the doc is edited after install; the marginal value over a PR-time review of a single eight-line constant is small while the surface area added is substantial. The F020 profile architecture rejected the same install-time-generation alternative for the same reasons (ADR-F020-005 plaintext-profiles-lock).
