# ADR-F023-001: Temp-ID format — `Ftmp-<8-char-hex>` from `secrets.token_hex(4)`

**Date:** 2026-05-21
**Status:** Accepted

**Context:** F023 defers final F-ID allocation to `/build` Step 7c. Features need a branch-local identifier that: (a) is collision-free in practice across all parallel branches in a single org, (b) is visually and programmatically distinguishable from final `F<NNN>` sequential IDs, (c) is terse enough to appear in directory names and git tag paths without becoming unwieldy, and (d) requires no external service or coordination. The identifier must also be greppable by `/metrics` and audit tooling so in-flight features can be tracked and later mapped to their final `F<NNN>` via `state.yaml.id_history`.

**Decision:** Temp IDs take the form `Ftmp-<8-char-hex>` where the hex portion is generated by `secrets.token_hex(4)` (4 bytes = 8 hex characters, drawn from a cryptographically secure RNG). Example: `Ftmp-3a7fc291`. The `Ftmp-` prefix is fixed and deliberately chosen: it is distinct from the `F<NNN>` sequential form, is grep-safe (no regex metacharacters), and sorts lexicographically separate from final IDs. 4 bytes of entropy gives approximately 4.3 billion possible values; the birthday-problem 50% collision threshold is ~65k allocations. At the operational rate of single-digit new features per developer per week, the expected time to first collision is measured in millennia — operationally collision-free without a central registry. On the astronomical chance of a collision (pre-existing dir with the same token), `allocate-temp` retries up to 3 times before exiting non-zero (EC-001).

**Consequences:** *Positive:* `Ftmp-` is a stable, greppable literal — `/metrics`, audit scripts, and log scanners can identify in-flight features with a single `grep 'Ftmp-'` without a schema-aware parser. The 8-char hex length is compact enough for directory names and git tag paths (e.g., `etc/feature/Ftmp-3a7fc291/build/phase-0/start`) while remaining human-scannable. Zero new dependencies — `secrets.token_hex` is Python 3.11+ stdlib. *Negative:* `/metrics` sees a mixed stream of `Ftmp-` phase tags and `F<NNN>` release tags for the same feature; it must join through `state.yaml.id_history` to produce unified rollups. Mixed tag streams are the direct cost of deferring allocation.

**Alternatives considered:**

- **Pure UUID (`Ftmp-01HGW3K7VQXJB...`):** 128 bits of entropy is overkill for a branch-local identifier that races only within a single developer's filesystem. The 26-character ULID/UUID suffix bloats directory names and git tag paths; tag paths must stay human-scannable per F021 BR-008's audit-log discipline. Rejected.

- **Shorter hex (4 chars = 2 bytes):** 2 bytes = ~65k possible values; birthday-problem 50% collision threshold at ~256 allocations. A team of four developers filing two features per week each would expect a collision inside three months. Not collision-free at any plausible operational scale. Rejected.

- **Longer hex (16 chars = 8 bytes):** 8 bytes = ~1.8 × 10^19 possible values. Adds no operational value over 4 bytes given the single-digit weekly allocation rate. The extra 8 characters in every directory name and git tag path impose pure readability cost with zero collision-resistance benefit. Rejected.

- **Hashids-style identifier (`Ftmp-jVbXa`):** Requires a third-party `hashids` library or equivalent. Introduces a new dependency for zero entropy benefit — hashids are a bijective encoding of integers, not a source of randomness. Rejected on the stdlib-only constraint (mirrors F021's `scripts/diagnostic_evidence.py` discipline).
