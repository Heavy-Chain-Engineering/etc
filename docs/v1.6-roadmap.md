### Tooling and infrastructure

- **`scripts/test-hook-prompt.py` — local fixture runner for prompt hooks.**
  Closes the prompt-test loop without requiring a Claude Code restart.
  Originated from the v1.5.1/1.5.2 harness-feedback debacle where three
  in-session attempts to fix a prompt hook all failed because settings are
  cached at session start. The runner reads `dist/settings-hooks.json`,
  extracts a named prompt, runs it via the Anthropic SDK against synthetic
  session fixtures, and asserts response shape. Single highest-leverage
  unblock for any future prompt-hook work.

- **`tasks.py score --feature` flag.** Surfaced during the `/hotfix` build:
  `score` does not honor `--feature` even though `bulk-create`, `waves`, and
  `tree` (partially) do. One-line fix.

- **`tasks.py tree --feature` filter.** Surfaced same build: `tree` shows all
  tasks across all features, not just the active one. Add `--feature` flag
  matching the other commands.

- **`/spec` writes a `state.yaml` at Phase 5.** Surfaced during `/hotfix`'s
  `/build` Step 1a: the validate step looks for a state.yaml with
  `classification: well-specified` to rubber-stamp specs that came through
  `/spec`, but `/spec`'s Phase 5 only writes `spec.md`, `gray-areas.md`, and
  `research/`. Adding state.yaml lets `/build` skip the inline DoR re-check.

- **Promote progress-capture pattern into `/spec`** as a standard Phase 1
  output. The `spec/.drafts/{slug}-progress.md` pattern we ad-hoc invented
  for `/hotfix` survived a Claude Code restart and let the resume session
  pick up exactly where the previous one paused, with all decisions intact.
  Worth promoting from improvisation into a standard `/spec` artifact for
  any spec that might span sessions.

### Architecture and observability (from Böckeler / Fowler harness-engineering article)

- **Architecture fitness functions** (ArchUnit-style). Etc has
  `check-invariants.sh` and `phase-gate` but nothing that mechanically
  enforces architectural constraints like "the domain layer never imports
  from the infrastructure layer." This is the middle category in Böckeler's
  three-regulation taxonomy (maintainability → architecture fitness →
  behaviour) and currently our weakest. Cataloging the constraints we care
  about, deciding which are mechanically checkable, and building a
  `check-fitness.sh` hook is the v1.6 work.

- **Drift detection as a continuous sensor.** Dead code, coverage rot,
  dependency staleness. Etc has no skill or hook that runs on a schedule
  looking for the harness getting weaker over time. Probably a `/drift`
  skill that runs weekly via the existing scheduler.

- **Harness coherence tooling / `/harness-review` skill.** Böckeler
  explicitly notes "we lack tools to reason about harness systems
  holistically." `compile-sdlc.py` validates the DSL syntactically but
  produces no coverage report — e.g., "of your 14 gates, 10 are guides and
  4 are sensors; you have 0 gates covering the 'behaviour' regulation
  category; gate X and gate Y are in tension because both fire on
  Edit|Write and gate X blocks what gate Y requires." Building this audit
  tool is both a real contribution and a forcing function for us to notice
  our own weaknesses.

- **Adopt `guides vs sensors` and `computational vs inferential` vocabulary
  in the README.** Re-label the 14 gates with `[G-c]` (computational guide),
  `[G-i]` (inferential guide), `[S-c]` (computational sensor), `[S-i]`
  (inferential sensor). ~30 minutes of doc work, makes the harness legible
  for the first time.

### Project-specific knowledge graph for VenLink-scale projects

- **Origin:** 2026-04-15. Jason's observation: "I'm VERY curious how we
  could leverage a project-specific knowledge graph to maintain knowledge.
  The VenLink project is a perfect example; I have 1,000,000+ lines of
  Salesforce export + thousands of pages of research."
- **The problem:** Some projects have so much domain context that fitting
  it into agent prompts via `inject-standards.sh` or `requires_reading`
  task fields is impractical. VenLink has 1M+ LOC of Salesforce export
  data, thousands of pages of regulatory and research documents, and the
  total context exceeds any single agent's window even with summarization.
  Today's harness assumes context can be enumerated as files; this breaks
  down at the scale where context IS a knowledge base.
- **Possible shape:** A per-project `.etc_sdlc/knowledge/` directory
  containing a graph database (something embeddable like Kuzu or an SQLite
  property-graph schema), populated by an indexer agent that reads the raw
  source material once and produces queryable nodes/edges. Subagents at
  spawn get a "knowledge graph access" tool that lets them query the graph
  by entity name, relationship, or natural-language question, returning
  the relevant subgraph as context. This is fundamentally different from
  RAG retrieval because the graph encodes RELATIONSHIPS (this regulation
  cites this section of that statute, this Salesforce object inherits
  from that one) not just semantic similarity.
- **Open questions:**
  - Which graph DB? Kuzu (embeddable, Cypher), SQLite-graph extensions,
    Memgraph, Neo4j (heavier)?
  - Indexer is a one-time job or continuous? How is freshness detected?
  - Does the graph live in `.etc_sdlc/` (per-project) or in a shared
    location across consumer projects?
  - How does this interact with `inject-standards.sh` which currently
    injects FILE contents at agent spawn? The graph access is a TOOL,
    not a context injection.
  - What's the bootstrapping cost (time, tokens, dollars) to index 1M+
    LOC of Salesforce data?
  - How do we test a graph-backed agent? The ground truth is the source
    documents, not the graph state.
- **Proposed first step:** A research spike PRD via `/spec` that surveys
  embeddable graph DBs against the requirements (Cypher-or-similar query
  language, no separate server process, fast on large inserts, Python
  bindings), then a small pilot in a fixture project with synthetic
  domain data, then a real pilot in VenLink with a narrow slice of the
  Salesforce export.
- **This is potentially v1.7 or v1.8 work, not v1.6** — it's a research
  spike before it's a buildable feature.
