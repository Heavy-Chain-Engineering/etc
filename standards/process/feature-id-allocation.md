# Feature ID Allocation

Every feature that passes through the etc SDLC carries a single human-readable
date-based identifier from `/spec` allocation through release. This standard
defines the format, the timing, the audit-trail schema in `state.yaml`, and the
backward-compatibility contract for legacy feature names.

## Status: MANDATORY

## Applies to: skills (/spec, /architect, /build), scripts/feature_id.py, /metrics

Applies to all features authored after the 2026-05-22 revision (see ADR
`docs/adrs/F-2026-05-22-feature-id-naming-revision-001-date-based-format.md`).
Legacy features (F001–F026) keep their existing `F<NNN>` names and are not
retroactively affected (see Forward-Only).

## REVISION 2026-05-22 (supersedes F023 ADR-001..004)

This standard was originally authored for F023's `Ftmp-<8-char-hex>` temp-ID
format with a temp→final rename at `/build` Step 7c.0. That format is
SUPERSEDED. The current scheme:

**Format:** `F-YYYY-MM-DD-<slug>` where the date is the current UTC date at
`/spec` allocation time, and `<slug>` is the operator-provided kebab-case
feature name.

**Examples:**
- `F-2026-05-22-python-installer-rewrite` (typical)
- `F-2026-05-22-fix-payment-bug` (short slug)
- `F-2026-05-22-fix-bug-2` (auto-suffix on same-day same-slug collision)

**Properties:**
- The dir name IS the feature_id (no separate `-<slug>` suffix segment as
  with the legacy F<NNN>-<slug> form).
- Cross-machine collision-safe: two operators on different machines pick
  different slugs near-trivially; the date+slug combo is near-unique.
- Chronologically sortable: `ls features/active/` shows features in UTC
  creation order via lexicographic sort.
- No temp→final rename at `/build` Step 7c.0 — the date-based form is
  final by construction. `resolve_final_id` is a no-op for these IDs.

**Collision behavior:** same-day same-slug → auto-suffix `-2`, `-3`, ...
up to a 99-attempt safety bound. Operationally rare.

**Backward compatibility:** the legacy `Ftmp-<8-char-hex>` form and the
sequential `F<NNN>` form are still recognized by readers
(`resolve_feature_path`, `resolve_final_id`'s short-circuit paths) for
any in-flight features from the F021-F026 era. No new feature is
produced in either legacy form.

The original F023 documentation below is preserved for archaeological
context only. The CURRENT contract is what appears above.

## Two ID forms (LEGACY — F023 era; see REVISION above)

**Temp form: `Ftmp-<hex>`**

- Format: `Ftmp-` prefix followed by exactly 8 lowercase hex characters
  (e.g., `Ftmp-3a9f1b2c`).
- Generated by `scripts/feature_id.py allocate-temp` at `/spec` Phase 2 Step 0.
- In force from that moment through `/build` Step 7c.
- Visible in: the active feature directory name
  (`.etc_sdlc/features/active/Ftmp-<hex>-<slug>/`), ADR filenames under `docs/adrs/`
  while the feature is in flight, `state.yaml.id_history`, and phase git tags
  (see Phase-N tag preservation below).
- Collision-free in practice: 4 bytes of entropy ≈ 4.3 billion possible values.

Per ADR-F023-001 (`docs/adrs/F023-001-temp-id-format.md`), the `Ftmp-` prefix was
chosen because it is greppable, human-readable, and visually distinct from the final form.

**Final form: `F<NNN>`**

- Format: `F` prefix followed by a zero-padded three-digit sequential integer
  (e.g., `F042`). Sequential allocation is handled by `scripts/feature_id.py
  allocate-next` (POSIX-atomic per F006 AC-002).
- Allocated by `scripts/feature_id.py resolve-final-id` at `/build` Step 7c,
  immediately before the active→shipped directory move.
- In force from Step 7c onward — including the release tag, the shipped directory
  name, ADR filenames after rename, and all `/metrics` rollup keys.
- Operators see `F<NNN>` in release notes, `/metrics` output, and the `shipped/`
  directory; the `Ftmp-<hex>` form is an internal-during-development detail.

Per ADR-F023-002 (`docs/adrs/F023-002-rename-at-step-7c.md`), rename at Step 7c
(not earlier) is the correct timing: it defers the only allocation that races to the
moment when conflict resolution is already the operator's lane (PR merge).

## Audit trail — state.yaml.id_history

`state.yaml` carries a top-level field `id_history` that records every ID assignment for
the feature. Schema:

```yaml
id_history:
  - form: temp
    value: Ftmp-3a9f1b2c
    written_at: 2026-05-21T14:00:00Z
  - form: final
    value: F042
    written_at: 2026-05-21T14:35:12Z
```

Rules:
- `form` is either `temp` or `final`.
- `value` is the literal ID string as allocated (e.g., `Ftmp-3a9f1b2c` or `F042`).
- `written_at` is an ISO-8601 UTC timestamp.
- The `temp` entry is written when `allocate-temp` creates the feature directory.
- The `final` entry is appended when `resolve-final-id` completes.
- The list is **append-only**: existing entries are never mutated or deleted.

`state.yaml.id_history` is the **canonical mapping** that `/metrics` reads when
reconciling phase git tags written under the `Ftmp-<hex>` form with their final
`F<NNN>` rollup key. Any `/metrics` computation that aggregates per-feature data MUST
join through this field. If a shipped feature's `state.yaml` contains no `id_history`
entry, the feature predates F023 and its directory name is already the final form.

## Phase-N tag preservation

Phase git tags written during `/build` waves use the temp form at the time of writing:

```
etc/feature/Ftmp-3a9f1b2c/build/phase-0/start
etc/feature/Ftmp-3a9f1b2c/build/phase-0/done
etc/feature/Ftmp-3a9f1b2c/build/phase-1/start
...
```

These tags are **append-only and permanent**. The harness does NOT delete, retag, or
force-update them at release time. The release tag uses the final form:

```
etc/feature/F042/release
```

Per ADR-F023-004 (`docs/adrs/F023-004-phase-tags-preserved-in-temp-form.md`), this
preserves the F021 BR-008 append-only tag discipline without exception. Phase tags are an
audit record of what was visible at the time the work ran; rewriting them would corrupt
that record. `/metrics` reconciles the two namespaces through `state.yaml.id_history`
(see above).

## Lifecycle — when each form appears

| Phase | Action | ID form |
|---|---|---|
| `/spec` Phase 2 Step 0 | `allocate-temp` creates feature dir | `Ftmp-<hex>` |
| `/spec` remainder | Writes `spec.md`, `design.md` | `Ftmp-<hex>` |
| `/architect` | Writes ADRs under `docs/adrs/Ftmp-<hex>-NNN-*.md` | `Ftmp-<hex>` |
| `/build` waves 0–N | Phase git tags written | `Ftmp-<hex>` |
| `/build` Step 7c.0 | `resolve-final-id` renames dir + ADRs, appends `id_history[final]` | transition |
| `/build` Step 7c.1 | Active→shipped directory move | `F<NNN>` |
| `/build` Step 7a | Release tag written | `F<NNN>` |
| `shipped/` directory | Final resting name | `F<NNN>` |
| `/metrics` rollups | Feature key | `F<NNN>` (mapped via `id_history`) |

The ADR rename at Step 7c uses `git mv` because `docs/adrs/` is git-tracked. Per
ADR-F023-003 (`docs/adrs/F023-003-adrs-renamed-via-git-mv.md`), `git mv` is mandatory
here (not `shutil.move`); it preserves rename history so reviewers see the
`Ftmp-<hex>-NNN-*.md` → `F<NNN>-NNN-*.md` transition explicitly in `git log --follow`.

## Forward-only

This convention applies to features whose `/spec` run occurs after the F023 release tag
lands. The first such feature is F024.

Legacy features (F001–F023) retain their existing `F<NNN>` names. Their `state.yaml`
files are not migrated; if `id_history` is absent, the feature predates F023. No
retroactive renaming is performed.

When `resolve-final-id` receives a directory whose name already matches `F<NNN>` form
(a pre-F023 feature being built), it exits 0 with a note on stderr: "feature already
has final ID; no rename needed." This preserves backward compatibility for any legacy
feature entering `/build` after the F023 release tag.

## Background

F023 was introduced to eliminate ID-collision bugs that arise when two developers run
`/spec` on separate branches without visibility into each other's allocations. Both
receive the same "next free" `F<NNN>` from the sequential allocator; the collision
surfaces only at PR-merge time, by which point both branches have committed work, written
ADRs, dispatched subagents, and written phase git tags — making recovery manual and lossy.

The load-bearing architectural decisions are recorded in four ADRs:

- **ADR-F023-001** (`docs/adrs/F023-001-temp-id-format.md`) — why `Ftmp-<8-char-hex>`
  (4-byte entropy, greppable prefix, collision-free in practice)
- **ADR-F023-002** (`docs/adrs/F023-002-rename-at-step-7c.md`) — why rename at Step 7c
  rather than at `/spec` or `/architect`
- **ADR-F023-003** (`docs/adrs/F023-003-adrs-renamed-via-git-mv.md`) — why `git mv`
  for ADRs and `shutil.move` for the feature directory (different tracking status)
- **ADR-F023-004** (`docs/adrs/F023-004-phase-tags-preserved-in-temp-form.md`) — why
  phase tags stay in `Ftmp-` form and `/metrics` bridges via `id_history`

This standard documents WHAT the convention is and WHERE each form appears. It does not
specify HOW allocation or rename is implemented; those details live in
`scripts/feature_id.py` and the four ADRs above.

## Anti-patterns

**Manually editing `state.yaml.id_history`.**
`id_history` is written exclusively by `scripts/feature_id.py`. Manual edits risk
mismatched timestamps, missing entries, or broken `/metrics` reconciliation. If you need
to repair a corrupted `id_history`, use `resolve-final-id --repair` (if available) and
record the remediation in the feature's `state.yaml` as a comment. Never silently
overwrite existing entries.

**Renaming the `Ftmp-<hex>` feature directory before Step 7c.**
Early rename bypasses the `id_history` append and leaves phase git tags in an orphaned
`Ftmp-` namespace with no mapping to the final `F<NNN>`. The rename MUST happen via
`resolve-final-id` at Step 7c, not by hand or by any earlier automation.

**Deleting or force-updating phase tags at release.**
Phase tags are append-only (F021 BR-008). Deleting or retagging them corrupts the audit
record and breaks any `/metrics` computation that relies on phase-tag timestamps. The
release tag (`etc/feature/F<NNN>/release`) is separate from the phase tags; writing the
former does not require touching the latter.

**Using `shutil.move` for ADR renames.**
`docs/adrs/` is git-tracked. Using `shutil.move` instead of `git mv` creates an
untracked rename that appears as a delete + add in `git diff`, losing rename history.
Always use `git mv` (argv-style `subprocess.run(["git", "mv", ...])`) for any file
under `docs/adrs/`.

**Skipping the `allocate-temp` call and creating the feature directory manually.**
`resolve-final-id` operates on directory shape and will still produce a correct rename.
However, the initial `temp` entry in `state.yaml.id_history` will be missing (it is
written by `allocate-temp`, not by `resolve-final-id`). This leaves a one-entry
`id_history` that may confuse `/metrics` tooling. Use `allocate-temp` for all new
features.
