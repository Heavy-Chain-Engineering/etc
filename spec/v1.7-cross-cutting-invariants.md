# PRD: Cross-Cutting Concept Registry

## Summary

The invariants framework (`standards/process/invariants.md` +
`hooks/check-invariants.sh`) enforces single-context rules: "no circular
imports in this package," "all models have created_at." It cannot express
cross-boundary semantic consistency: "when bounded context A says
'org_id', it means the SAME thing as when bounded context B says
'org_id.'"

Three independent signals confirm this gap:

1. **VenLink audit** -- tenant isolation broken because `org_id` meant
   "vendor's org" in one context and "client's org" in another. STATUS_MAP
   used internal terms while Salesforce export used different vocabulary.
   Global mutable `_handlers` dict had no lifecycle contract across
   contexts.

2. **NC System retrospective (260K LOC)** -- IAM permission vocabulary
   was inconsistent across bounded contexts. CRUD-style permissions in
   one module, domain-labeled permissions in another. No enforcement
   caught it because no single-file analysis can detect cross-boundary
   drift.

3. **Jim Snyder (Chief Architect, CS DISCO) DbC recommendation** --
   vocabulary lists catch naming drift but not behavioral assumptions.
   Preconditions define what the caller must provide, postconditions
   define what the provider guarantees, and dependency direction prevents
   consumers from inventing expectations.

This PRD extends `INVARIANTS.md` with a new entry category --
`CONCEPT-NNN` -- that declares cross-boundary semantic contracts with
Design by Contract fields, vocabulary registries, and verify commands
that scan across multiple bounded contexts. A new verify-phase hook
(`check-concepts.sh`) runs these slower cross-boundary checks separately
from the fast per-edit `check-invariants.sh`.

## Scope

### In Scope

- `CONCEPT-NNN` entry format in `INVARIANTS.md` (backward-compatible
  with `INV-NNN` parse pattern)
- Design by Contract fields: Precondition, Postcondition, Invariant
  (natural-language, testable assertions)
- Vocabulary registry sub-format with context-to-term mapping tables
- `hooks/check-concepts.sh` -- new verify-phase hook that parses
  CONCEPT entries and runs their verify commands
- `scripts/check-vocabulary.py` -- vocabulary verify script that loads
  vocabulary tables and checks context-specific term usage
- Compiler validation of CONCEPT entry syntax in `compile-sdlc.py`
- At least 3 VenLink-derived example concepts
- DSL registration of `check-concepts.sh` in `spec/etc_sdlc.yaml`
- Tests for the new hook, vocabulary script, and compiler validation

### Out of Scope

- **Automated concept discovery.** Detecting that two contexts use the
  same term differently without a human declaring it is a research
  problem, not a v1.7 deliverable.
- **Graph-based concept relationship modeling.** That is the knowledge
  graph roadmap item.
- **Runtime enforcement of DbC contracts.** Test-time and verify-time
  only.
- **Auto-generation of test_concepts.py from CONCEPT entries.** The
  compiler currently copies and validates; adding Python codegen is a
  significant capability expansion deferred to a future PRD.
- **Changes to check-invariants.sh.** The existing per-edit hook stays
  fast and unchanged. CONCEPT checks run at verify phase only.
- **Separate CONCEPTS.md file.** CONCEPT entries live in `INVARIANTS.md`
  alongside INV entries, maintaining one cascading file with one
  discovery mechanism.

## Requirements

### BR-001: CONCEPT-NNN entry format

`INVARIANTS.md` MUST support entries with `## CONCEPT-NNN:` headings
(where NNN is a zero-padded 3-digit number) alongside existing
`## INV-NNN:` headings. A CONCEPT entry MUST contain these fields:

```markdown
## CONCEPT-001: Short description of the cross-boundary concept

Prose explanation of what this concept means and why it must be
consistent across boundaries.

- **Contexts:** Comma-separated list of bounded contexts that use this concept
- **Precondition:** What the caller/consumer must provide (natural language)
- **Postcondition:** What the provider/producer guarantees (natural language)
- **Invariant:** The property that must always hold across all listed contexts
- **Layers:** Comma-separated enforcement layers (test, hook, agent-instructions, ci)
- **Verify:** `shell command that returns non-empty output on VIOLATION`
- **Fail action:** Block merge | Warn
```

### BR-002: CONCEPT entry backward compatibility

The `CONCEPT-NNN` heading format MUST follow the same parse pattern as
`INV-NNN`: a level-2 heading (`##`) followed by an ID and colon-separated
description. The **Verify** and **Fail action** fields use identical
syntax to INV entries. This ensures any tooling that generically parses
`INVARIANTS.md` headings and verify lines can handle both entry types
with minimal regex extension (`INV-` becomes `(?:INV|CONCEPT)-`).

### BR-003: Design by Contract fields

Every CONCEPT entry MUST include **Precondition**, **Postcondition**,
and **Invariant** fields. These are natural-language assertions that
are human-readable and testable, not formal specification language.
The assertions follow Jim Snyder's DbC structure:

- **Precondition**: obligation of the caller -- what must be true before
  the concept is used. Maps to "what does the consumer provide?"
- **Postcondition**: obligation of the provider -- what is guaranteed
  after the concept is used. Maps to "what does the producer deliver?"
- **Invariant**: the property that holds at all times across all listed
  contexts. Maps to "what must always be true regardless of caller?"

The dependency direction principle applies: if context A depends on
context B's concept, context B owns the contract.

### BR-004: Contexts field semantics

The **Contexts** field lists bounded contexts as human-readable names
(e.g., "IAM, Relationships, Compliance, Search"). Each context name
SHOULD correspond to a top-level directory, module, or package in the
consumer project's source tree. The verify command uses these context
names to scope its file scanning.

### BR-005: Vocabulary registry sub-format

CONCEPT entries MAY include a vocabulary sub-section using this format:

```markdown
### Vocabulary: canonical_name
| Context | Term | Canonical |
|---------|------|-----------|
| Context A | "term_in_a" | canonical_term |
| Context B | "term_in_b" | canonical_term |
- **Verify:** `python3 scripts/check-vocabulary.py canonical_name`
```

The vocabulary table declares, per bounded context, what terms that
context uses and what canonical term each maps to. The verify command
checks that each context's source code uses only the terms declared
for that context in the table.

### BR-006: check-concepts.sh hook script

A new hook script `hooks/check-concepts.sh` MUST:

1. Parse all `INVARIANTS.md` files in the project (using the same
   cascading discovery as `check-invariants.sh`: project root +
   component ancestors of the current file).
2. Extract `## CONCEPT-NNN:` headings and their `**Verify:**` commands
   using the same regex pattern as `check-invariants.sh` uses for
   `## INV-NNN:`, extended to match `CONCEPT-` prefix.
3. Run each verify command from the project root.
4. Report violations to stderr with the format:
   `CONCEPT VIOLATION [CONCEPT-NNN]: Verify command found violations:`
   followed by up to 5 lines of output.
5. Exit 2 if any violations are found, exit 0 otherwise.
6. Support the `# concept-exempt: CONCEPT-NNN` per-line exemption
   pattern (parallel to `# invariant-exempt: INV-NNN`).

The script MUST follow the same structure, stdin parsing, and error
handling patterns as `hooks/check-invariants.sh`.

### BR-007: check-concepts.sh runs at verify phase only

`check-concepts.sh` MUST be registered in `spec/etc_sdlc.yaml` as a
**Notification** event hook (not PreToolUse), running at the verify
phase of the SDLC pipeline. CONCEPT verify commands scan multiple
directories and are inherently slower than single-file INV checks.
They MUST NOT run on every Edit/Write -- that would degrade the
per-edit hook latency that v1.6 optimized.

If no suitable Notification event exists in the current DSL for
verify-phase execution, register it as a `PostToolUse` hook on
`Task` tool calls (which occur at phase transitions), or document the
exact DSL gate definition needed.

### BR-008: check-vocabulary.py script

A new script `scripts/check-vocabulary.py` MUST:

1. Accept one argument: the vocabulary name (e.g., `vendor_status`).
2. Find the vocabulary table in the nearest `INVARIANTS.md` by
   searching for `### Vocabulary: {name}`.
3. Parse the markdown table to extract context-term-canonical triples.
4. For each context, scan source files in the context's directory for
   usage of terms NOT declared in that context's vocabulary rows.
5. Print violations to stdout (non-empty = failure, per the invariants
   convention).
6. Exit 0 on success (no violations), exit 1 on failure.
7. Respect `# concept-exempt: CONCEPT-NNN` comment patterns.

The script MUST handle:
- Context directories that don't exist (warning, not error)
- Empty vocabulary tables (pass -- nothing to check)
- Terms that are substrings of other terms (exact word-boundary matching)

### BR-009: Compiler validation of CONCEPT entries

`compile-sdlc.py` MUST validate CONCEPT entries in `INVARIANTS.md`
during compilation. Validation checks:

1. Every `## CONCEPT-NNN:` heading has a **Contexts** field (required
   for cross-boundary concepts; without it, the entry is just an INV).
2. Every CONCEPT entry has **Precondition**, **Postcondition**, and
   **Invariant** fields.
3. Every CONCEPT entry has a **Verify** field with a non-empty command.
4. Every CONCEPT entry has a **Fail action** field.
5. CONCEPT IDs are unique and do not collide with INV IDs in the same
   file.

Validation failures MUST be reported to stderr and cause a non-zero
exit code. The compiler does NOT execute verify commands -- it only
validates syntax.

### BR-010: DSL registration

`spec/etc_sdlc.yaml` MUST include a gate definition for
`check-concepts.sh`. The gate definition MUST specify:

- `event`: the appropriate lifecycle event for verify-phase execution
- `type`: command
- `script`: check-concepts.sh
- `timeout`: 30 (seconds -- CONCEPT checks scan more files than INV checks)
- `description`: "Cross-boundary concept consistency verification"

### BR-011: Per-line exemption pattern

CONCEPT entries MUST support per-line exemption via the comment pattern
`# concept-exempt: CONCEPT-NNN`. This is parallel to the existing
`# invariant-exempt: INV-NNN` pattern. The exemption comment MUST
cause `check-concepts.sh` and `check-vocabulary.py` to skip that line
when scanning for violations.

### BR-012: Fail action semantics

CONCEPT entries support two fail actions:

- **Block merge**: the verify-phase hook reports the violation and
  blocks the merge/completion. This is the default for security and
  data-integrity concepts.
- **Warn**: the verify-phase hook reports the violation but does not
  block. This is for concepts being phased in or where violations
  need triage before enforcement.

**Block edit** is NOT supported for CONCEPT entries because they run
at verify phase, not per-edit.

## Example CONCEPT Entries (VenLink-Derived)

These three examples are derived from real VenLink bugs. They
demonstrate the format and would have caught the corresponding
production issues.

### Example 1: Tenant Ownership (VenLink Finding 5)

```markdown
## CONCEPT-001: Organization ownership for multi-tenant queries

All database queries that return tenant-scoped data MUST filter by the
authenticated user's organization_id, not the target entity's
organization_id. The VenLink search bug occurred because VendorProfile
was indexed under the vendor's org_id, but search filtered by the
client's org_id -- same concept name, different semantic meaning.

- **Contexts:** IAM, Relationships, Compliance, Search
- **Precondition:** Authenticated user has a non-null organization_id
- **Postcondition:** Every row in the result set belongs to an org the
  user has access to (via direct membership or relationship grant)
- **Invariant:** `query.filter_org == auth_user.org_id` (never `entity.org_id`)
- **Layers:** test, hook, agent-instructions
- **Verify:** `grep -rn 'filter.*org_id' src/ | grep -v 'current_user' | grep -v '# concept-exempt: CONCEPT-001'`
- **Fail action:** Block merge
```

### Example 2: Vendor Status Vocabulary (VenLink Finding 4)

```markdown
## CONCEPT-002: Vendor status vocabulary across ETL and API boundaries

The canonical vocabulary for vendor lifecycle status must be declared
per context. The VenLink STATUS_MAP bug occurred because the ETL context
used Salesforce export terms ("Approved", "Onboarding In Process") while
the internal API used different terms ("active", "suspended"). The
mapping was incomplete and silently dropped unrecognized statuses.

- **Contexts:** Salesforce ETL, Internal API, Compliance
- **Precondition:** Status values entering the system are one of the
  declared terms for the originating context
- **Postcondition:** Status values stored in the database use canonical
  terms, not context-specific synonyms
- **Invariant:** No status value exists in the database that is not in
  the canonical vocabulary
- **Layers:** test, ci
- **Verify:** `python3 scripts/check-vocabulary.py vendor_status`
- **Fail action:** Block merge

### Vocabulary: vendor_status
| Context | Term | Canonical |
|---------|------|-----------|
| Salesforce ETL | "Approved" | active |
| Salesforce ETL | "Onboarding In Process" | pending |
| Salesforce ETL | "Terminated" | inactive |
| Salesforce ETL | "Suspended" | suspended |
| Internal API | "active" | active |
| Internal API | "pending" | pending |
| Internal API | "inactive" | inactive |
| Internal API | "suspended" | suspended |
| Compliance | "compliant" | active |
| Compliance | "non_compliant" | suspended |
```

### Example 3: Event Handler Lifecycle (VenLink Finding 3)

```markdown
## CONCEPT-003: Event handler registration lifecycle

Event handlers must be registered via an explicit registration function
call, not by mutating module-level state at import time. The VenLink
_handlers bug occurred because one context registered handlers by
mutating a module-level dict during import, while another context
expected handlers to be registered via register_search_event_handlers().
Neither approach had a lifecycle contract.

- **Contexts:** Events, Search, Notifications
- **Precondition:** The application has completed initialization before
  any event handler is invoked
- **Postcondition:** All registered handlers are callable and their
  registrations are reversible (for testing)
- **Invariant:** No event handler registration occurs at module import
  time; all registration happens in explicit setup functions
- **Layers:** test, hook, agent-instructions
- **Verify:** `grep -rn '_handlers\[' src/ | grep -v 'def register' | grep -v '# concept-exempt: CONCEPT-003'`
- **Fail action:** Block merge
```

## Acceptance Criteria

1. **CONCEPT entries parse alongside INV entries.** An `INVARIANTS.md`
   file containing both `## INV-001:` and `## CONCEPT-001:` entries
   is valid. The compiler validates both. `check-invariants.sh`
   continues to parse and enforce only INV entries. `check-concepts.sh`
   parses and enforces only CONCEPT entries. [BR-001, BR-002, BR-006]

2. **check-concepts.sh detects violations.** A test that: (a) creates
   a fixture `INVARIANTS.md` with a CONCEPT entry whose verify command
   finds violations, (b) runs `check-concepts.sh`, (c) asserts exit
   code 2 and stderr contains `CONCEPT VIOLATION [CONCEPT-NNN]`.
   [BR-006]

3. **check-concepts.sh passes clean projects.** A test that: (a)
   creates a fixture with a CONCEPT entry whose verify command
   produces empty output, (b) runs `check-concepts.sh`, (c) asserts
   exit code 0. [BR-006]

4. **Exemption pattern works.** A test that: (a) creates a file with
   a line that would violate CONCEPT-001, (b) adds
   `# concept-exempt: CONCEPT-001` to that line, (c) runs the verify
   command, (d) asserts the exempted line is not in the output.
   [BR-011]

5. **check-vocabulary.py detects term violations.** A test that: (a)
   creates a vocabulary table for `test_status` with context A
   allowed terms `["active", "pending"]`, (b) creates a source file
   in context A's directory containing `"suspended"`, (c) runs
   `check-vocabulary.py test_status`, (d) asserts non-empty output
   naming the violation. [BR-008]

6. **check-vocabulary.py passes clean vocabularies.** A test that: (a)
   creates a vocabulary table, (b) ensures all source files use only
   declared terms, (c) asserts empty output. [BR-008]

7. **Compiler validates CONCEPT syntax.** A test that: (a) creates an
   `INVARIANTS.md` with a CONCEPT entry missing the Contexts field,
   (b) runs compiler validation, (c) asserts a non-zero exit and
   error message naming the missing field. [BR-009]

8. **Compiler accepts valid CONCEPT entries.** A test that creates a
   well-formed CONCEPT entry and asserts the compiler validates it
   without error. [BR-009]

9. **DbC fields are present and parseable.** A test that parses a
   CONCEPT entry and asserts Precondition, Postcondition, and
   Invariant fields are extracted as non-empty strings. [BR-003]

10. **DSL gate registered.** After compilation, `dist/settings-hooks.json`
    includes the `check-concepts.sh` hook at the configured event.
    [BR-010]

11. **VenLink CONCEPT-001 would have caught the tenant bug.** A test
    that creates a fixture file containing `filter(org_id=vendor.org_id)`
    (without `current_user`) and runs CONCEPT-001's verify command,
    asserting the line is flagged. [Example 1]

12. **VenLink CONCEPT-002 would have caught the STATUS_MAP bug.** A
    test that creates a fixture where the Salesforce ETL context uses
    a term not in its vocabulary table and runs `check-vocabulary.py`,
    asserting a violation. [Example 2]

13. **check-invariants.sh is unchanged.** The existing hook script
    is byte-identical before and after this feature. All existing
    tests in invariant-related test files pass without modification.
    [scope guarantee]

14. **Full test suite green.** `python3 -m pytest -q` passes with
    the new tests added. [completeness]

15. **Compile clean.** `python3 compile-sdlc.py spec/etc_sdlc.yaml`
    succeeds. Gate count increments by 1 (the new check-concepts gate).
    [BR-010]

## Edge Cases

1. **INVARIANTS.md with only CONCEPT entries, no INV entries.**
   Valid. `check-invariants.sh` finds no INV entries and exits 0.
   `check-concepts.sh` finds CONCEPT entries and runs their verify
   commands. Both hooks coexist without interference.

2. **CONCEPT verify command that takes >30 seconds.** The DSL gate
   timeout (30s) kills the hook. The violation is NOT reported, and
   the hook exits with the timeout error. This is correct behavior --
   verify commands should be fast (grep-based). A slow verify command
   is a bug in the concept definition, not in the hook.

3. **Vocabulary table with zero rows.** `check-vocabulary.py` treats
   an empty table as "no constraints" and passes. This allows concepts
   to be declared before their vocabulary is fully populated.

4. **Context directory that does not exist.** `check-vocabulary.py`
   emits a warning to stderr but does not fail. Missing context
   directories are common during early project setup.

5. **CONCEPT-NNN and INV-NNN with the same number.** Valid. The
   prefixes distinguish them. `CONCEPT-001` and `INV-001` are
   different entries. The compiler validates uniqueness within each
   prefix, not across prefixes.

6. **Nested INVARIANTS.md files with CONCEPT entries at different
   levels.** Both `check-concepts.sh` and `check-invariants.sh` use
   the same cascading discovery (project root + component ancestors).
   CONCEPT entries at component level add to project-level concepts;
   they never override or relax them. Same additive semantics as INV.

7. **Verify command that uses `check-vocabulary.py` but the script
   is not installed.** The verify command fails with a non-zero exit
   code. `check-concepts.sh` treats command failures as "could not
   verify" (warning to stderr), not as "violation found." This
   prevents a missing script from blocking all merges.

8. **Term in vocabulary table is a substring of another term.**
   `check-vocabulary.py` MUST use word-boundary matching (e.g.,
   `\bterm\b` regex) to avoid false positives where "active" matches
   "inactive."

## Security Considerations

### Verify command injection via INVARIANTS.md

CONCEPT verify commands are shell strings executed via `eval` in
`check-concepts.sh`, identical to the existing `check-invariants.sh`
pattern. The same trust model applies: `INVARIANTS.md` is a
repository-controlled file, and its contents are trusted at the same
level as any other checked-in script. No additional attack surface
beyond what `check-invariants.sh` already has.

### check-vocabulary.py file scanning

The vocabulary script scans source directories listed in the Contexts
field. These directories are within the project root (enforced by the
hook's `CWD` scoping). No path traversal risk beyond what `grep -rn`
in verify commands already has.

### Exemption comment abuse

A developer could add `# concept-exempt: CONCEPT-001` to every line
to suppress all violations. This is the same risk as
`# invariant-exempt: INV-001`. Mitigation is the same: code review
catches excessive exemptions, and the exemption pattern is visible in
diffs. No new mitigation needed beyond existing process.

## Research Notes

### Codebase findings

- `check-invariants.sh` line 87 uses regex
  `^##[[:space:]]+(INV-[0-9]+):` to match invariant headings. The
  CONCEPT extension requires changing this to
  `^##[[:space:]]+((?:INV|CONCEPT)-[0-9]+):` in the NEW script only.
  The existing script is untouched.

- `check-invariants.sh` line 92 uses regex
  `\*\*Verify:\*\*[[:space:]]*\`(.+)\`` to extract verify commands.
  This regex is format-agnostic (works for both INV and CONCEPT entries)
  and can be reused directly in `check-concepts.sh`.

- `compile-sdlc.py` currently has no INVARIANTS.md validation. The
  `validate_concepts()` function is a new addition, not a modification
  of existing validation. It should be called from `main()` after
  the existing compilation steps.

- The `getting-started/INVARIANTS.md` example project already has 4
  INV entries. Adding CONCEPT examples there demonstrates coexistence.

- `compile_scripts()` in `compile-sdlc.py` (line 437) already copies
  all files from `scripts/` to `dist/scripts/`. The new
  `check-vocabulary.py` will be picked up automatically.

### Design by Contract (Meyer) -- applied pragmatically

Bertrand Meyer's DbC defines preconditions (caller's obligation),
postconditions (supplier's guarantee), and class invariants (always-true
properties). The formal Eiffel syntax (`require`, `ensure`, `invariant`
blocks) is not appropriate for a markdown registry. This PRD adopts
the SEMANTIC structure (three fields per entry) with NATURAL LANGUAGE
assertions rather than formal notation. The verify command serves as
the executable enforcement -- it is the postcondition check that the
system can actually run.

### DDD Ubiquitous Language (Evans) -- applied to concept registry

Eric Evans defines bounded context as "a boundary where we eliminate
any kind of ambiguity." The CONCEPT entry format enforces Evans'
principle by requiring explicit Contexts listing, canonical definitions,
and vocabulary tables that map context-specific terms to canonical
terms. This is the ubiquitous language registry that DDD recommends
but rarely sees implemented as a machine-verifiable artifact.

### Cross-boundary consistency patterns

Schema registries in microservice architectures serve the same
function: a single authority for data structures across service
boundaries. The CONCEPT registry is the harness equivalent -- a
single-file authority for semantic concepts across bounded contexts,
with the added benefit of verify commands that enforce the registry
automatically.

### Gray area resolutions

1. **Same file vs. separate file**: CONCEPT entries live in
   `INVARIANTS.md`, not a separate `CONCEPTS.md`. Rationale: reuses
   existing cascading file discovery in `check-invariants.sh` pattern.
   One file, one discovery mechanism, two parsers (one per hook).

2. **Extend check-invariants.sh vs. new check-concepts.sh**: New hook.
   Rationale: CONCEPT verify commands scan multiple directories and
   are inherently slower. The brief explicitly flags this. Per-edit
   `check-invariants.sh` must stay fast (v1.6 hook-cost-reduction
   was specifically about reducing per-edit hook latency).

3. **Vocabulary verify: Python vs. bash**: Python. Rationale: parsing
   markdown tables, matching terms with word-boundary regex per
   context directory is complex enough that bash would be fragile.

4. **DbC syntax: formal vs. pragmatic**: Pragmatic (natural language).
   Rationale: the verify command is the executable contract. Formal
   notation adds ceremony without enforcement power.

5. **Compiler auto-generation of test_concepts.py**: No. Rationale:
