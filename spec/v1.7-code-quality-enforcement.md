# PRD: Automated Code Quality Enforcement Pipeline (v1.7)

## Summary

The `standards/code/` and `standards/testing/` libraries contain ~35 rules across 7 documents, but every rule is enforced at the "agent reads the doc" level. The VenLink audit (2026-04-16) proved this is insufficient: 10 code quality findings, 7 mechanically detectable, all shipped because compliance was a function of LLM attention span. This spec adds two new enforcement tiers to the harness — ruff-based linting (Level 1) and AST-analysis hooks (Level 2) — so that the standards docs become the specification and the tooling becomes the enforcement.

**VenLink motivation:** Mid-module imports in 3 files, a global mutable `_handlers` defaultdict that caused 30 minutes of test isolation debugging, a no-op function that only logged, exception names missing `Error` suffix across 4 classes, Python 2-style except syntax in 5 files, and a fragile test asserting on a global dict that required 3 fix attempts. Findings 1, 2, 4, 5, and 6 would have been caught by the enforcement pipeline specified here.

## Scope

### In Scope

- Ruff configuration audit mapping every standard rule to its ruff equivalent (or documenting "no ruff rule")
- Reference ruff config (`standards/code/ruff-reference.toml`) that consumer projects copy or extend
- Enforcement annotations on all rules in `standards/code/*.md` and `standards/testing/*.md`
- Two new custom hooks for non-lintable patterns (global mutable state, no-op function detection)
- One new hook script `hooks/check-code-quality.sh` with Python helpers
- Three new standards docs: `import-discipline.md`, `test-isolation.md`, `fixture-fidelity.md`
- Compiler `--audit-enforcement` flag on `compile-sdlc.py`
- Ruff audit document at `standards/code/ruff-audit.md`
- Gate registration in `spec/etc_sdlc.yaml`
- Test coverage for all new code

### Out of Scope

- Changing ruff itself (we consume it, not maintain it)
- CI pipeline integration (consumer project responsibility)
- Language-specific standards beyond Python (future: TypeScript, Go)
- Nesting depth detection hook (follow-up; no ruff rule, lower VenLink priority)
- Test naming convention hook (follow-up; lower priority than mutable state and no-op detection)
- Automated fix/autofix behavior (this spec detects and blocks; fixing is the agent's job)

### Level 2: Custom Hooks

#### BR-010: check-code-quality.sh hook script

Create `hooks/check-code-quality.sh` as a PreToolUse hook for Edit|Write operations. The hook:
1. Accepts the same JSON input format as `check-invariants.sh` (`tool_input.file_path`, `cwd`)
2. Runs sub-checks via Python helper scripts in `hooks/helpers/`
3. Reports violations on stderr with check ID, file, line, and description
4. Exits 0 (pass) or 2 (block with violation details)
5. Only analyzes `.py` files (skip non-Python edits silently)

#### BR-011: Global mutable state detection (CQ-001)

Create `hooks/helpers/check_mutable_globals.py` that:
1. Parses a Python file using the `ast` module
2. Detects module-level assignments of mutable types: `list()`, `[]`, `dict()`, `{}`, `set()`, `defaultdict()`, `deque()`
3. Exempts `__init__.py` files (legitimate re-export patterns)
4. Exempts assignments with `Final` type annotation
5. Exempts assignments in `if TYPE_CHECKING:` blocks
6. Prints violations as `CQ-001:{line}:{col}: Module-level mutable assignment: {variable_name} = {type}`
7. Exits 0 if no violations, 1 if violations found

This addresses VenLink finding number 2 (venlink.common.events._handlers defaultdict).

#### BR-012: No-op function detection (CQ-002)

Create `hooks/helpers/check_noop_functions.py` that:
1. Parses a Python file using the `ast` module
2. Detects functions whose body consists entirely of:
   - Only `logger.*()` calls (any logging method)
   - Only `pass` statements
   - Only docstring + `pass`
   - Only docstring + `logger.*()` calls
3. Exempts abstract methods (decorated with `@abstractmethod`)
4. Exempts protocol methods (in classes inheriting `Protocol`)
5. Exempts `__init__` methods (legitimate empty inits)
6. Exempts functions decorated with `@pytest.fixture`
7. Prints violations as `CQ-002:{line}:{col}: No-op function (body is only logging): {function_name}`
8. Exits 0 if no violations, 1 if violations found

This addresses VenLink finding number 3 (register_search_event_handlers that only logs).

#### BR-013: Gate registration for check-code-quality

Register `code-quality-check` gate in `spec/etc_sdlc.yaml` under `gates:`:

- event: PreToolUse
- matcher: "Edit|Write"
- type: command
- script: check-code-quality.sh
- timeout: 10
- on_failure: block
- description: Detects non-lintable code quality violations via AST analysis: module-level mutable state and no-op functions.

## Edge Cases

### EC-001: Files with legitimate module-level mutables
`__init__.py` files, `Final`-annotated assignments, and `TYPE_CHECKING` blocks are exempted. All other module-level mutables are surfaced as warnings. The agent can add `# cq-exempt: CQ-001 -- <reason>` to suppress.

### EC-002: Functions that are intentionally log-only
Legitimate log-only functions (e.g., audit trail emitters) can add `# cq-exempt: CQ-002 -- audit trail` to suppress. The exemption comment is the documentation.

### EC-003: Circular import breaks
E402 fires for mid-module imports. The standard documents that circular breaks require a comment `# circular-break: <module> -> <module>`. E402 can be suppressed per-line with `# noqa: E402` when the circular break comment is present. The `--audit-enforcement` flag does not validate per-line noqa -- that is ruff's job.

### EC-004: Third-party stubs without type annotations
The typing-standards.md already allows `type: ignore[import-untyped]` for third-party stubs. ANN rules only fire on the project's own code, not on imported modules.

### EC-005: Non-Python files edited
`check-code-quality.sh` silently passes (exit 0) when the file being edited is not a `.py` file.

### EC-006: Empty or malformed Python files
The AST parser may fail on files being created (empty) or mid-edit (syntax errors). The hook MUST catch `SyntaxError` from `ast.parse()` and pass silently -- blocking on unparseable files would prevent the agent from creating new files incrementally.

## Technical Constraints

1. **Python 3.11+ required** for `ast` module features used by helper scripts.
2. **No external dependencies** for hook helpers -- only stdlib (`ast`, `sys`, `pathlib`, `json`). Hooks must work without a virtualenv.
3. **Hook timeout: 10 seconds** -- AST parsing of a single file is fast (under 100ms); the timeout is generous.
4. **Reference config is advisory** -- consumer projects copy and extend it. The harness does not modify consumer `pyproject.toml` files.
5. **Existing ruff configs in root and platform `pyproject.toml` are not modified** -- the reference config is a separate file that documents the recommended rules.
6. **compile-sdlc.py must remain dependency-light** -- it already uses `yaml`; adding `tomllib` (stdlib in 3.11+) for TOML parsing is acceptable.

## Security Considerations

1. **No code execution in hooks beyond ast.parse()** -- the helpers parse but never eval or exec file contents.
2. **Hook input sanitization** -- `check-code-quality.sh` validates `file_path` from JSON input before passing to Python helpers; rejects paths containing `..` or absolute paths outside the project.
3. **No secrets in enforcement annotations** -- annotations reference ruff rule codes and hook names, never credentials or tokens.
