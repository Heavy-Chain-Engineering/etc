# PRD: v1.8 Observability and Memory

## Motivation

Smith (ATTCKDigital/smith) demonstrates four patterns worth pulling into
etc: automatic session logging, structured parked-ideas directory,
proactive accumulated-learnings ledger, and queued autonomous task
processing. Each fills a real gap in etc:

1. **No session audit trail.** etc has SessionStart hooks (compaction-
   recovery, reinject-context) but never logs what happens *in* a session.
2. **Roadmap is ad-hoc.** `docs/v1.6-roadmap.md` grew organically and is
   already hard to scan. No structure, no lifecycle, no way to promote a
   parked item to a spec.
3. **Learnings are reactive.** `.etc_sdlc/antipatterns.md` captures
   failures. Nothing captures positive learnings or mid-stream discoveries
   (e.g., "Kuzu was archived" — learned this session, nowhere to record it
   systematically).
4. **No overnight queue.** Claude Code provides `CronCreate`, `/schedule`,
   `/loop`, and `EnterWorktree` primitives. etc doesn't compose them into
   a reusable "queue tasks for overnight processing" pattern.

Metrics tracking (also in Smith) is explicitly out of scope — Jason uses
Claude Code via OAuth, not the API, so per-token cost tracking isn't
actionable.

## Scope

### In scope

## Feature A — Session logger hooks

### BR-A1: `session-start-logger.sh`

A new SessionStart hook script that, when fired:

1. Creates `.etc_sdlc/sessions/` if it doesn't exist
2. Generates a session log at
   `.etc_sdlc/sessions/{YYYY-MM-DD}-{HHMMSS}-{slug}.md` where `{slug}`
   is derived from the first user prompt (or `startup` if no prompt yet)
3. Writes initial frontmatter: session ID, started_at, working directory,
   git branch, git HEAD sha
4. Exits 0 (never blocks)
5. Stores the active session path in `.etc_sdlc/sessions/.active`
   (single-line file with the absolute path) so other hooks and skills
   can append to it

### BR-A2: `session-end-review.sh`

A new Stop hook script that, when fired:

1. Reads `.etc_sdlc/sessions/.active` to find the current session log
2. If no active session, exits 0 silently
3. Appends a "Session End" section containing:
   - `ended_at` timestamp
   - `duration_seconds`
   - Git diff stat against the session's starting HEAD (files changed,
     additions, deletions)
   - Count of commits made during the session (current HEAD vs start HEAD)
4. Removes `.etc_sdlc/sessions/.active`
5. Exits 0 (never blocks)

### BR-A3: Session log append helper

A shared helper (bash function or separate script) that other hooks can
call to append entries to the active session log. Initial callers:

- `mark-dirty.sh` — log when a source file is modified
- (Future) `check-required-reading.sh` — log blocks
- (Future) `ci-gate.sh` — log failures

### BR-A4: Session log format

Machine-readable Markdown, one entry per event:

```markdown
---
session_id: 2026-04-17-143022-session-logger
started_at: 2026-04-17T14:30:22Z
cwd: /Users/jason/src/etc-system-engineering
branch: main
head_at_start: abc1234
---

# Session Log

## 2026-04-17T14:30:22Z — session started

## 2026-04-17T14:32:15Z — file edited
- file: hooks/ci-gate.sh
- tool: Edit

## Session End
- ended_at: 2026-04-17T15:42:11Z
- duration_seconds: 4309
- files_changed: 3
- commits: 1
```

### BR-A5: Gate registration

Both hooks registered in `spec/etc_sdlc.yaml`. The SessionStart hook
runs with `matcher: "startup"` (fires at genuine session start, not
compaction).

### BR-A6: Gitignore strategy

`.etc_sdlc/sessions/` is gitignored by default (per-user, per-session
ephemeral data). Add an explicit `.gitignore` entry. Sessions can grow
unbounded; this is persistent-per-laptop, not persistent-per-team.

## Feature B — Roadmap pattern

### BR-B1: `.etc_sdlc/roadmap/` directory convention

Each roadmap item is a single Markdown file at
`.etc_sdlc/roadmap/{NNN}-{slug}.md` where `NNN` is a three-digit
zero-padded sequence. Files have frontmatter:

```markdown
---
id: ROADMAP-001
slug: knowledge-graph-poc
status: parked | active | specced | building | shipped | abandoned
priority: P0 | P1 | P2 | P3
created_at: 2026-04-17T00:00:00Z
promoted_at: null
tags: [kg, research, venlink]
---

# ROADMAP-001: Knowledge graph POC for domain-heavy projects

Prose description of the idea, scope hints, references, etc.

## Why

Why this matters.

## References

- spec/kg-poc.md (if promoted to spec)
- memory/project-closed-loop-ticket-pipeline.md
```

Status transitions:

- `parked` → `active` when prioritized for near-term work
- `active` → `specced` when a PRD exists at `spec/{slug}.md`
- `specced` → `building` when `/build` is running
- `building` → `shipped` when the commit/tag lands on main
- Any → `abandoned` when explicitly dropped; requires a `### Abandoned`
  section explaining why

### BR-B2: `/roadmap` skill

New skill `skills/roadmap/SKILL.md` supporting subcommands:

- `/roadmap list [--status=STATUS] [--priority=P0|P1|P2|P3]` — table of
  entries filtered by status or priority
- `/roadmap add "description"` — Socratic capture: asks 3 questions
  (why does this matter? what's the hint of scope? what's the priority?),
  writes a new `NNN-{slug}.md` file, returns the path
- `/roadmap show ROADMAP-NNN` — print the file
- `/roadmap promote ROADMAP-NNN` — transition parked → active, then
  offer to kick off `/spec` with the roadmap content as the brief
- `/roadmap abandon ROADMAP-NNN "reason"` — transition to abandoned with
  a mandatory reason

### BR-B3: One-time migration of `docs/v1.6-roadmap.md`

The existing `docs/v1.6-roadmap.md` has ~8 in-flight items and ~6 ideas-
to-evaluate. Migrate each into `.etc_sdlc/roadmap/{NNN}-{slug}.md` with
appropriate status. Preserve original prose. Leave `docs/v1.6-roadmap.md`
with a pointer to the new location, or delete it if empty post-migration.

### BR-B4: Gitignore strategy

`.etc_sdlc/roadmap/` is **committed** (unlike sessions). Roadmap is team-
shared artifact, not per-laptop ephemera.

## Feature C — Ledger pattern

### BR-C1: `.etc_sdlc/ledger.md` append-only file

A single file with a strict format. Entries are append-only; never
edit existing entries. Format:

```markdown
# etc Engineering Ledger

Append-only log of accumulated learnings, discoveries, and positive
patterns. Different from antipatterns.md — this captures what we
learned, not just what we got wrong.

---

## 2026-04-17 — Kuzu graph database archived
**Category:** discovery
**Source:** session 2026-04-17-093015-kg-poc
**Lesson:** Don't lock in specific tooling at spec time unless you have
strong justification. Kuzu was archived in late 2025; the KG POC PRD
originally recommended it. Fix: specify evaluation criteria, let the
implementer pick against current data.

---

## 2026-04-17 — Content-hash primitive shows up everywhere
**Category:** learning
**Source:** session 2026-04-17-093015-kg-poc
**Lesson:** A single well-chosen primitive (sha256-of-content → marker
file) appears in five different places: per-subagent hook caching,
graph staleness detection, extraction cache invalidation, source-file
change detection, rebuild-equivalence proof. When you catch yourself
reaching for the same primitive repeatedly, it's signal it's the right
one.
```

### BR-C2: `/ledger` skill

New skill `skills/ledger/SKILL.md`:

- `/ledger show [--category=CAT] [--since=DATE]` — print filtered entries
- `/ledger add` — Socratic capture: asks category (learning | discovery |
  antipattern | positive-pattern), source, lesson. Appends to ledger.
- `/ledger search QUERY` — grep across ledger entries

### BR-C3: `/retrospective` integration

The existing `/retrospective` skill, on completion, offers to append
the top 3 learnings to the ledger. This is an *optional* prompt — user
can decline.

### BR-C4: Gitignore strategy

`.etc_sdlc/ledger.md` is **committed**. Team-shared learning artifact.

## Feature D — Queue skill

### BR-D1: `.etc_sdlc/queue/` directory convention

Each queued task is a markdown file at
`.etc_sdlc/queue/{NNN}-{slug}.md` with frontmatter:

```markdown
---
id: QUEUE-001
slug: index-venlink-sources
status: pending | running | done | failed
isolation: worktree | current
max_runtime_minutes: 60
created_at: 2026-04-17T10:15:00Z
started_at: null
completed_at: null
result_path: null  # Populated when done
---

# QUEUE-001: Index VenLink source material for KG POC

## Task

Full prompt for the autonomous run. What to do, what success looks like,
what to commit.

## Acceptance

Concrete completion criteria.
```

### BR-D2: `/queue` skill

New skill `skills/queue/SKILL.md` supporting:

- `/queue list` — table of queued/running/done tasks
- `/queue add "description"` — Socratic capture: asks for task
  description, isolation mode (default worktree), max runtime. Writes a
  QUEUE-NNN file.
- `/queue process [ID]` — picks the next pending task (or specific ID),
  uses `EnterWorktree` for isolation (if requested), dispatches the
  task, captures the result, updates status, writes `result_path`.
- `/queue status ID` — show status of one task
- `/queue schedule` — creates a cron trigger via `CronCreate` to run
  `/queue process` daily at a user-chosen time (default 02:00 local)

### BR-D3: Isolation default

Queued tasks default to `isolation: worktree`. A separate worktree is
created per task to prevent interference with in-progress work and to
cleanly capture results on a branch.

### BR-D4: Failure handling

If a queued task fails (max runtime exceeded, uncaught error, subagent
returns with `continue: false`), its status transitions to `failed` and
the error is captured in a `## Failure` section of the QUEUE file. The
next `/queue process` run skips `failed` tasks unless they're
explicitly re-queued via `/queue retry ID`.

### BR-D5: Gitignore strategy

`.etc_sdlc/queue/` is **gitignored**. Queued tasks are operator-
specific; they're inputs to overnight automation, not team artifacts.
Once a task transitions to `done`, the operator can optionally copy
the result to a tracked location (a commit, a ledger entry, a spec) —
but the queue file itself is ephemeral.

## Edge cases

1. **Slug collision in session log naming** — two sessions started
   within the same second → append a `-2` suffix and increment.
2. **Git not initialized** — session logger tries to read HEAD, fails
   → frontmatter writes `head_at_start: (no git)` and continues.
3. **`/roadmap add` during `.etc_sdlc/roadmap/` absence** — skill creates
   the directory.
4. **Sequence number exhaustion** — roadmap IDs are 3-digit zero-padded,
   supporting up to 999. Beyond that, allow 4-digit. Not a near-term
   concern.
5. **Ledger file missing** — `/ledger add` creates it with the canonical
   header prose.
6. **Queue task with no worktree available** — if `EnterWorktree` fails,
   the task stays `pending` and the error is surfaced; it is not
   silently skipped.
7. **Queue task times out** — transition to `failed` with "max runtime
   exceeded" in the Failure section.
8. **`.etc_sdlc/sessions/.active` orphaned** — if a previous session
   died without running the end hook, the next SessionStart overwrites
   `.active`; this is fine.
9. **Migration of `docs/v1.6-roadmap.md`** — preserve original prose
   per item, assign status based on "In-flight" vs "Ideas to evaluate"
   sections, default priority P2.

## Security considerations

- **Session logs may contain sensitive prompts.** They're gitignored
  and live on-laptop. Operator responsibility.
- **Queue tasks run autonomously.** They must be dispatched with the
  same safety guardrails as interactive builds (safety-guardrails hook,
  tier-0 preflight). Adversarial or misspecified queue tasks must be
  blocked by the same primitives that block malicious main-thread work.
- **Session end review captures git diff.** If a session modifies
  `.env*` or credential files, the diff would include sensitive
  content. The session end hook MUST redact any path matching
  `*.env*`, `credentials*`, `*secret*` from the diff summary.

## Research notes

- Smith's session logger was the direct inspiration for Feature A;
  their pattern is validated by active daily use. Port the idea, not
  the code — Smith is MIT but we want to fit etc's idioms.
- Smith's scheduler is a macOS LaunchAgent because Claude Code didn't
  provide cron primitives at Smith's initial release. Claude Code now
  provides `CronCreate`, so `/queue schedule` is a thin wrapper rather
  than a LaunchAgent install. This is a major simplification.
- Smith's vault is flatter than etc's `.etc_sdlc/`. Don't port the
  structure; fit into etc's.
- The v1.6-roadmap.md file has been functioning as a roadmap ad-hoc.
  This PRD finally formalizes what it's been trying to be.
