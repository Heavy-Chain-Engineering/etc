# PRD: Integration Seam Test Evidence Gate

## Summary

The etc harness enforces that tests PASS but not that integration tests EXIST for the cross-boundary seams that matter. A project can achieve 98% unit test coverage with zero integration tests at bounded-context boundaries, and the harness gives it a green checkmark. This is not hypothetical: the VenLink audit (2026-04-16) found architecturally broken search tenant isolation, STATUS_MAP never validated against real data, and dev endpoints diverging from production paths -- all because service-level unit tests passed while cross-boundary integration tests did not exist. The NC System retrospective (2026-03-12, 260K LOC) identified "missing e2e tests (only had service-level mocks)" as the number one integration gap.

This feature introduces a **seam manifest** (`SEAMS.md`) that declares which bounded context pairs exchange data, a **three-level evidence system** (L1: test exists, L2: test exercises both sides, L3: test uses real dependencies), and a **verify-phase check** (`check-seam-evidence.sh`) that structurally verifies declared seam evidence before a build is declared done. Progressive adoption ensures consumer projects are not blocked: no `SEAMS.md` means the check skips with a warning, and only declared seams are enforced.

The seam evidence gate is an architectural fitness function -- an automated check that verifies structural properties of the codebase's integration test coverage, not behavioral correctness. It answers the question: "for every integration boundary the architect declared important, does a test exist that exercises that boundary?"

## Scope

### In Scope

- `SEAMS.md` manifest format: machine-parseable markdown declaring integration seams between bounded contexts, following the same structural convention as `INVARIANTS.md`.
- Evidence level definitions (L1, L2, L3) with concrete, mechanical verification for each level.
- `check-seam-evidence.sh` hook script: parses `SEAMS.md`, checks evidence at the declared level for each seam, returns exit codes.
- Compiler support: `compile-sdlc.py` validates `SEAMS.md` syntax and reports seam count in the compile summary.
- Verify-phase gate registration in `spec/etc_sdlc.yaml`.
- Progressive adoption path: no `SEAMS.md` = skip + warn (exit 0).
- `SEAM-DEV` entries for fixture fidelity (dev simulation seams).
- Integration with the `/build` verify step.
- Integration with the `/spec` skill: PRDs for cross-boundary features should declare seams.

### Out of Scope

- Automated seam DISCOVERY (detecting cross-boundary imports that lack seam declarations). This is the `/drift` skill from the roadmap.
- Test generation (the gate checks evidence exists, not writes it).
- CI pipeline integration (consumer project responsibility).
- Runtime seam monitoring (observability, not testing).
- Cross-cutting concept registry (`CONCEPTS.md` / Gap 2) -- SEAM entries may optionally reference concepts, but no validation of concept references is performed by this feature.
- Behavioral test correctness ("is the test right?") -- only structural evidence ("does the test exist and run?").

## Requirements

### BR-001: SEAMS.md file format

Every consumer project MAY have a `SEAMS.md` file at its root. The file is human-readable markdown that is also machine-parseable. Each seam is a level-2 heading with the following structure:

```markdown
## SEAM-001: Short description of the integration seam
- **Producer:** path/to/producer/directory/
- **Consumer:** path/to/consumer/directory/
- **Interface:** Description of the contract (model, API endpoint, event, etc.)
- **Integration test:** path/to/tests/integration/test_file.py
- **Evidence level:** L1 | L2 | L3
- **Critical path:** User-facing flow that depends on this seam
```

Required fields: ID (in heading), Producer, Consumer, Interface, Integration test, Evidence level. Optional fields: Critical path, Concept, Constraint (for SEAM-DEV entries).

The ID format is `SEAM-NNN` for standard seams and `SEAM-DEV-NNN` for fixture fidelity seams.

### BR-002: Evidence level L1 -- Test exists

For every seam declared at evidence level L1, the check verifies:

1. The file declared in `**Integration test:**` exists on disk.
2. The file contains the `@pytest.mark.integration` marker (or a `pytestmark = [pytest.mark.integration]` module-level assignment).

L1 is the minimum evidence: the test file exists and is tagged as an integration test. It does not verify the test's content or imports.

### BR-003: Evidence level L2 -- Test exercises both sides

For every seam declared at evidence level L2, the check verifies everything in L1, plus:

3. The test file contains an import from the **Producer** directory (any module under the declared producer path).
4. The test file contains an import from the **Consumer** directory (any module under the declared consumer path).

The import check uses grep for `from {producer_module}` or `import {producer_module}` where `{producer_module}` is derived from the producer path by converting `/` to `.` and stripping `src/` prefix if present. The check looks for the top-level package of the producer and consumer, not an exact module path.

### BR-004: Evidence level L3 -- Test uses real dependencies

For every seam declared at evidence level L3, the check verifies everything in L2, plus:

5. The test file contains the marker comment `# seam-evidence: L3-real` on a line by itself.
6. The test file does NOT contain mock/patch calls targeting the producer or consumer modules. Specifically, the check greps for `mock.patch.*{producer_package}`, `@patch.*{producer_package}`, `MagicMock.*{producer_package}`, `monkeypatch.setattr.*{producer_package}` (and the same for `{consumer_package}`). If any match is found, the check fails for that seam with a diagnostic naming the offending line.

Mocking of UNRELATED modules (e.g., `@patch("stripe.Charge.create")` in a test for an IAM-to-Relationships seam) does NOT cause a failure. Only mocks targeting the declared producer or consumer packages are flagged.

### BR-005: SEAM-DEV entries for fixture fidelity

Seam entries with the `SEAM-DEV-NNN` ID prefix declare fixture fidelity seams: where a dev simulation or test fixture is the PRODUCER and a real service is the CONSUMER. These entries support an additional optional field:

```markdown
- **Constraint:** Human-readable invariant the dev simulation must satisfy
```

SEAM-DEV entries follow the same evidence level checks as standard SEAM entries. The `**Constraint:**` field is informational and is not mechanically verified by the check script (it is a code-review concern).

### BR-006: check-seam-evidence.sh script

A new bash script `hooks/check-seam-evidence.sh` that:

1. Reads `SEAMS.md` from the project root (determined from `$CWD` in the hook input).
2. If no `SEAMS.md` exists: prints a warning to stderr ("No SEAMS.md found -- seam evidence checks skipped"), exits 0.
3. Parses each `## SEAM-NNN:` or `## SEAM-DEV-NNN:` heading and extracts: ID, Producer, Consumer, Integration test path, Evidence level.
4. For each seam entry, runs the evidence checks corresponding to the declared level (L1, L2, or L3).
5. Collects all failures. After processing all entries, prints a summary to stderr.
6. Exit codes: 0 = all checks pass (or no SEAMS.md), 2 = one or more evidence checks failed.

The script follows the same structure as `check-invariants.sh`: read stdin for `{tool_input, cwd}` JSON, parse the manifest, run checks, report violations, exit with appropriate code.

### BR-007: Verify-phase gate registration

A new gate entry in `spec/etc_sdlc.yaml`:

```yaml
seam-evidence-check:
  description: >
    Validates integration seam evidence from SEAMS.md. For each declared
    seam, checks that the integration test exists, has the correct marker,
    exercises both sides (L2+), and uses real dependencies (L3). Runs at
    verify phase only -- not per-edit.
  event: Stop
  type: command
  script: check-seam-evidence.sh
  timeout: 30
  on_failure: block
```

The gate is registered on the `Stop` event (same as `ci-pipeline`) so it runs at the end of a build session. The `ci-pipeline` agent gate already runs tests; the seam evidence check runs in parallel and verifies the structural evidence exists.

Note: The gate uses the `Stop` event rather than a phase-specific trigger because the harness does not currently have a verify-phase-specific event. The script itself is lightweight enough (structural file checks, no test execution) to run at every session stop alongside the CI pipeline gate.

### BR-008: Compiler support

`compile-sdlc.py` is extended to:

1. Copy `hooks/check-seam-evidence.sh` to `dist/hooks/` (already handled by the existing `compile_gates` function that copies all gate scripts).
2. The new gate entry in `etc_sdlc.yaml` is compiled into `settings-hooks.json` automatically by the existing gate compilation logic.

No changes to the compiler's Python code are needed -- the existing gate compilation pipeline handles new command-type gates automatically. The new gate entry in the YAML is sufficient.

### BR-009: Progressive adoption

The adoption path has three stages:

1. **No SEAMS.md:** The check exits 0 with a stderr warning. No enforcement. The compiler reports "No SEAMS.md found" in its summary. The project is not blocked.
2. **SEAMS.md with entries:** The check enforces evidence for every declared seam. Undeclared seams are not checked (the manifest is opt-in per seam).
3. **Drift detection (future):** A `/drift` skill scans for cross-directory imports that lack corresponding seam entries and proposes new seam declarations. This is out of scope for this feature.

### BR-010: Integration with /spec

When the `/spec` skill writes a PRD for a feature that crosses bounded context boundaries, the PRD SHOULD include a "Seam Declarations" section listing the seams introduced or affected by the feature, with proposed evidence levels. This becomes a Phase 2 (Architecture Review) deliverable. The seam declarations in the PRD are advisory; the actual `SEAMS.md` entries are created during the Build phase.

### BR-011: Integration with /build verify step

After all waves complete, the `/build` verify step runs seam evidence checks in addition to the existing test suite. If any seam evidence check fails, the build is not marked complete. The agent must either (a) write the missing integration test, (b) update the evidence level to a lower tier, or (c) remove the seam declaration (with justification).

## Acceptance Criteria

1. **SEAMS.md with L1 seam passes when test exists.** Given a `SEAMS.md` with one `SEAM-001` entry at L1 pointing to `tests/integration/test_foo.py`, and that file exists and contains `@pytest.mark.integration`, `check-seam-evidence.sh` exits 0 with no violations. [BR-001, BR-002]

2. **L1 fails when test file missing.** Same setup but `tests/integration/test_foo.py` does not exist. The script exits 2 with a diagnostic naming `SEAM-001` and the missing file path. [BR-002, BR-006]

3. **L1 fails when integration marker missing.** The test file exists but has no `@pytest.mark.integration` or `pytestmark` assignment. The script exits 2. [BR-002]

4. **L2 passes when test imports both producer and consumer.** Given `SEAM-002` at L2 with producer `src/venlink/iam/` and consumer `src/venlink/relationships/`, and the test file contains `from venlink.iam import ...` and `from venlink.relationships import ...`, the check passes. [BR-003]

5. **L2 fails when test imports only producer.** The test file imports from `venlink.iam` but not from `venlink.relationships`. The script exits 2 with a diagnostic naming the missing consumer import. [BR-003]

6. **L3 passes with L3-real marker and no interface mocks.** Given `SEAM-003` at L3, the test file contains `# seam-evidence: L3-real` and does NOT contain `@patch("venlink.iam...")` or `mock.patch("venlink.relationships...")`. The check passes. [BR-004]

7. **L3 fails when L3-real marker is absent.** The test file passes L2 checks but has no `# seam-evidence: L3-real` comment. The script exits 2. [BR-004]

8. **L3 fails when producer module is mocked.** The test file has `# seam-evidence: L3-real` but also contains `@patch("venlink.iam.user_service.get_user")`. The script exits 2 with a diagnostic naming the specific mock line. [BR-004]

9. **L3 allows mocking of unrelated modules.** The test file mocks `stripe.Charge.create` (unrelated to producer/consumer). The check passes for the seam. [BR-004]

10. **No SEAMS.md exits 0 with warning.** When the project has no `SEAMS.md`, the script exits 0 and prints "No SEAMS.md found -- seam evidence checks skipped" to stderr. [BR-006, BR-009]

11. **SEAM-DEV entry with constraint field is parsed.** A `SEAM-DEV-001` entry with a `**Constraint:**` field is parsed without error. The constraint field is not mechanically verified. [BR-005]

12. **Multiple seams: partial failure reports all failures.** Given three seams where SEAM-001 passes, SEAM-002 fails L1, and SEAM-003 fails L3, the script exits 2 and the stderr output names both SEAM-002 and SEAM-003 with their specific failures. [BR-006]

13. **Module-level pytestmark assignment is recognized.** A test file with `pytestmark = [pytest.mark.integration]` instead of `@pytest.mark.integration` passes the L1 marker check. [BR-002]

14. **Compile produces the gate entry.** After `python3 compile-sdlc.py spec/etc_sdlc.yaml`, `dist/hooks/check-seam-evidence.sh` exists and is executable, and `dist/settings-hooks.json` contains the `seam-evidence-check` gate. Gates count increases by 1. [BR-007, BR-008]

15. **SEAMS.md format is parseable end-to-end.** A SEAMS.md with 5 entries (3 standard, 2 SEAM-DEV) across evidence levels L1/L2/L3 is parsed without error. All 5 entries are checked. [BR-001, BR-006]

## Edge Cases

1. **SEAMS.md exists but is empty (no seam entries).** The script exits 0 with a warning "SEAMS.md found but contains no seam entries." No failure -- an empty manifest is valid during progressive adoption.

2. **Integration test path contains spaces.** The file existence check must quote the path. The check uses `[[ -f "$test_path" ]]` with proper quoting.

3. **Producer and consumer paths overlap.** A seam where the producer is `src/venlink/` and the consumer is `src/venlink/relationships/` (consumer is a subdirectory of producer). The L2 import check should still require imports from BOTH paths. The consumer import must be from the more specific path.

4. **Two seam entries point to the same integration test.** Valid -- one test may exercise multiple seams. Each seam's evidence level is checked independently against the shared test file.

5. **Evidence level is not L1, L2, or L3.** The parser warns "Unknown evidence level 'L4' for SEAM-005 -- skipping" to stderr and does not fail the overall check. Forward-compatible with future evidence levels.

6. **The test file exists but is empty (0 bytes).** L1 fails (no integration marker found). L2 fails (no imports found). The diagnostic is specific: "SEAM-001: test file exists but has no @pytest.mark.integration marker."

7. **SEAMS.md has a seam entry with a missing required field.** The parser warns "SEAM-003: missing required field 'Evidence level' -- skipping" to stderr. The seam is skipped, not treated as a failure. This supports incremental authoring.

8. **Producer path uses `src/` prefix but import uses the package name without `src/`.** The import derivation strips the `src/` prefix when computing the expected package name. `src/venlink/iam/` becomes `venlink.iam` for import checking.

9. **The `# seam-evidence: L3-real` marker appears in a comment block, not on its own line.** The grep matches lines where the marker is the only non-whitespace content: `^\s*# seam-evidence: L3-real\s*$`. Inline comments like `x = 1  # seam-evidence: L3-real` do NOT match.

10. **Consumer project has both SEAMS.md and INVARIANTS.md.** Both files are checked by their respective hooks independently. No interaction between the two check scripts.

11. **Seam test file is in a subdirectory not on the Python path.** The check only verifies file existence and content (grep-based). It does not import the test or validate Python path configuration.

12. **Mock grep hits a comment line, not actual code.** Accepted as a false positive. The L3 check is structural, not semantic. If a comment line contains `@patch("venlink.iam.service")`, the check flags it. The developer can remove the comment or split the test. This is a deliberate simplicity tradeoff -- AST analysis would be correct but expensive and fragile.

## Security Considerations

### Command injection via SEAMS.md paths

The check script reads file paths from SEAMS.md and passes them to `[[ -f "$path" ]]` and `grep`. All paths MUST be quoted. The script MUST NOT use `eval` on any value read from SEAMS.md. Path values are used only in bash conditionals and as grep file arguments, never in command construction.

### Path traversal via producer/consumer paths

Producer and consumer paths are used for import-name derivation (string manipulation), not for file access. The integration test path IS used for file access. The script MUST verify that the resolved test file path is under the project root: `[[ "$(realpath "$test_path")" == "$CWD"* ]]`.

### Symlink following in test file paths

The file existence check uses `realpath` to resolve symlinks before checking the path prefix. A symlinked test file pointing outside the project root fails the path-traversal check.

### Denial of service via large SEAMS.md

A SEAMS.md with thousands of entries could slow the verify phase. The script has a configurable entry limit (default: 200). If exceeded, the script warns and checks only the first 200 entries. This is a safety valve, not an expected condition.

# Integration Seams

Declares integration boundaries between bounded contexts. Each seam is verified
by `check-seam-evidence.sh` during the verify phase. See
`standards/process/invariants.md` for the companion invariant registry.

## SEAM-001: IAM to Relationships (user org membership lookup)
- **Producer:** src/venlink/iam/
- **Consumer:** src/venlink/relationships/
- **Interface:** `UserOrganizationMembership` model query
- **Integration test:** tests/integration/test_iam_relationships.py
- **Evidence level:** L2
- **Critical path:** Invitation approval flow
- **Concept:** UserOrganizationMembership

## SEAM-002: Salesforce ETL to Internal Models (status mapping)
- **Producer:** src/venlink/integrations/salesforce/
- **Consumer:** src/venlink/relationships/models/
- **Interface:** STATUS_MAP dictionary mapping Salesforce status strings to internal enum values
- **Integration test:** tests/integration/test_salesforce_status_mapping.py
- **Evidence level:** L3
- **Critical path:** Vendor onboarding via Salesforce sync

## SEAM-003: IAM to Search (tenant-scoped query filtering)
- **Producer:** src/venlink/iam/
- **Consumer:** src/venlink/search/
- **Interface:** `current_user.organization_id` filter applied to all search queries
- **Integration test:** tests/integration/test_search_tenant_isolation.py
- **Evidence level:** L3
- **Critical path:** All search operations

## SEAM-004: Events to Relationship Service (handler registration)
- **Producer:** src/venlink/events/
- **Consumer:** src/venlink/relationships/event_handlers.py
- **Interface:** Explicit event handler registration via `register_handler()` call
- **Integration test:** tests/integration/test_event_registration.py
- **Evidence level:** L1
- **Critical path:** Async workflow triggers

- **Integration test:** tests/integration/test_dev_simulation_fidelity.py

## VenLink Seam Examples

These are the seams that, if declared with the specified evidence levels, would have caught the VenLink audit findings:

### Finding: Search tenant isolation (VenLink Finding #5)

**Seam:** `SEAM-003: IAM to Search`
- **Producer:** `src/venlink/iam/` (provides `current_user.organization_id`)
- **Consumer:** `src/venlink/search/` (must filter all queries by org)
- **Evidence level:** L3 required
- **What L3 catches:** An L3 test must use a real database with real user/org records and verify that search results are filtered by the calling user's organization. Mocking the IAM lookup would have hidden the filter bug -- the mock would return the "right" org ID, but the real code path had a missing WHERE clause. The L3 "no mocks of producer/consumer" requirement forces the test to exercise the actual IAM-to-Search data flow.

### Finding: STATUS_MAP validation (VenLink Finding #4)

**Seam:** `SEAM-002: Salesforce ETL to Internal Models`
- **Producer:** `src/venlink/integrations/salesforce/` (provides raw status strings)
- **Consumer:** `src/venlink/relationships/models/` (maps strings to internal enums)
- **Evidence level:** L3 required
- **What L3 catches:** An L3 test must load real Salesforce export data (a CSV fixture from a real Salesforce org, not invented data) and verify the STATUS_MAP covers all observed values. Mocking the Salesforce response would have hidden the mismatch -- the mock would only contain statuses the developer knew about, not the ones actually in the data.

### Finding: Dev simulation fidelity (VenLink Finding #9)

**Seam:** `SEAM-DEV-001: Dev simulation to Approval flow`
- **Producer:** `src/venlink/dev/routes.py::simulate_duplicate_vendor`
- **Consumer:** `src/venlink/relationships/invitation_service.py`
- **Evidence level:** L3 required
- **What L3 catches:** An L3 test importing from both the dev routes module and the invitation service would have revealed that `simulate_duplicate_vendor` creates a vendor org with zero admin users, which violates the approval flow's assumption that every vendor org has at least one admin. The `**Constraint:**` field in the SEAM-DEV entry makes this invariant visible at the manifest level before you even read the test.

## Research Notes

### Codebase findings

- **`hooks/check-invariants.sh`** is the direct model: 142 lines of bash, reads INVARIANTS.md, extracts verify commands via regex, runs them, collects violations, exits 0 or 2. The SEAMS.md parser follows the same line-by-line approach with `BASH_REMATCH` for field extraction.
- **`standards/testing/testing-standards.md`** defines Tier 2 (integration) with the `@pytest.mark.integration` marker. This is the marker the L1 check looks for. No new marker is introduced.
- **`standards/process/invariants.md`** defines the INVARIANTS.md format. SEAMS.md follows the same H2-heading-with-structured-fields convention but is a separate file with different field names and semantics.
- **`standards/process/definition-of-done.md`** checks "All tests pass" and "CI pipeline passes" but has no check for "integration tests exist for declared seams." This feature adds that check at the verify-phase level.
- **No existing `@pytest.mark.integration` usage** in the etc repo's own tests (all tests use `@pytest.mark.parametrize` only). The etc repo is a meta-project (harness tooling); integration markers are expected in consumer projects, not in the harness itself.
- **`compile-sdlc.py`** automatically handles new command-type gates: it copies the script to `dist/hooks/`, generates the `settings-hooks.json` entry, and reports the gate count. No Python code changes needed for a new command gate.
- **`spec/etc_sdlc.yaml`** has 15 gates (after v1.6). The new `seam-evidence-check` gate will be the 16th.
- **Existing hooks are all <10s.** The seam evidence check targets <5s for a typical SEAMS.md with <50 entries. The 30s timeout in the gate registration provides headroom.

### Web research findings

- **Contract testing (Pact model):** Consumer-driven contract testing verifies API compatibility between services. Pact generates contracts from consumer tests and verifies them against the provider. The SEAMS.md approach is lighter-weight: it does not generate or verify contracts programmatically, but it ensures that integration test files exist and exercise both sides of the boundary. Pact is complementary but out of scope for this feature.
- **Architecture fitness functions (Neal Ford / ThoughtWorks):** Fitness functions are automated checks that verify architectural characteristics. The seam evidence check is a fitness function: it verifies the structural property "every declared integration boundary has a test that exercises it." The key insight from fitness function literature is that the check should be automated, objective, and integrated into the development pipeline -- all properties this feature satisfies.
- **Martin Fowler's testing pyramid:** The pyramid recommends many unit tests, fewer integration tests, fewest E2E tests. The seam evidence gate does not change the pyramid -- it enforces that the middle layer (integration) is not empty for architecturally important boundaries. The problem it solves is not "too few integration tests" in general but "zero integration tests at specific critical boundaries."
- **"Integration test" definition varies:** Some teams use "integration test" to mean "test that hits a database." The seam evidence gate is specific: an integration test for a SEAM is one that exercises the boundary between two declared bounded contexts. The `@pytest.mark.integration` marker plus import checks (L2) pin the definition mechanically.

### Gap analysis

All requirements are fully specified from codebase patterns and the brief. No unfillable gaps. The only forward dependency (Gap 2 / CONCEPTS.md) is handled via the optional `**Concept:**` field with no validation.
